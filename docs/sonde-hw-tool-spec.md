## 1  KiCad File Format Reference

`sonde-hw` generates **KiCad 8** files (schema version `20231120`).
Both `.kicad_sch` and `.kicad_pcb` files use an S-expression format
with parenthesized nodes.

### 1.1  `.kicad_sch` root structure

```sexpr
(kicad_sch
  (version 20231120)
  (generator "sonde-hw")
  (generator_version "0.1.0")
  (uuid "<root-uuid>")
  (paper "A4")

  (title_block
    (title "sonde О“Г‡Г¶ <config-name>")
    (date "2026-01-01")
    (rev "0.1")
    (comment 1 "Config hash: <sha256>")
    (comment 2 "Generated by sonde-hw <version>")
  )

  (lib_symbols ...)   ; в”¬Вє 1.2
  (symbol ...)         ; в”¬Вє 1.3  (one per component instance)
  (wire ...)           ; в”¬Вє 1.4
  (label ...)          ; в”¬Вє 1.5
  (power_port ...)     ; в”¬Вє 1.5
  (global_label ...)   ; в”¬Вє 1.5
)
```

### 1.2  `lib_symbols` block

Every symbol referenced by a component instance must first be declared
inside `(lib_symbols ...)`.  The declaration copies the full symbol
definition from the KiCad standard library or from a custom library.

```sexpr
(lib_symbols
  (symbol "Device:R"
    (pin_names (offset 0))
    (in_bom yes) (on_board yes)
    (property "Reference" "R" ...)
    (property "Value" "R" ...)
    (property "Footprint" "" ...)
    (property "Datasheet" "~" ...)
    (symbol "R_0_1"
      (polyline (pts (xy 0 -1.016) (xy 0 1.016)) (stroke ...))
      ...
    )
    (symbol "R_1_1"
      (pin passive line (at 0 1.27 270) (length 0.508) (name "~" ...) (number "1" ...))
      (pin passive line (at 0 -1.27 90) (length 0.508) (name "~" ...) (number "2" ...))
    )
  )
  ;; ... additional symbols
)
```

Key rules:
* Every `(symbol "Lib:Name" ...)` under `lib_symbols` uses the
  *library-qualified* name (e.g. `"Device:R"`).
* Each symbol contains one or more *units* (`symbol "R_0_1"` for the
  body, `symbol "R_1_1"` for the pins).
* Pin numbers and types (passive, input, output, power_in, power_out,
  bidirectional, tri_state, unconnected) must match the library
  definitions exactly; mismatches cause ERC errors.

### 1.3  Component instances (`symbol`)

Each placed component is a top-level `(symbol ...)` node:

```sexpr
(symbol
  (lib_id "Device:R")
  (at 100.33 62.23 0)           ; x y rotation_degrees
  (unit 1)
  (exclude_from_sim no)
  (in_bom yes)
  (on_board yes)
  (dnp no)
  (uuid "a1b2c3d4-...")
  (property "Reference" "R1" (at 100.33 60.0 0) ...)
  (property "Value" "10kв•¬вЊђ" (at 100.33 64.5 0) ...)
  (property "Footprint" "Resistor_SMD:R_0402_1005Metric" (at ...) ...)
  (property "Datasheet" "~" (at ...) ...)
  (property "LCSC" "C25744" (at ...) ...)
  (pin "1" (uuid "..."))
  (pin "2" (uuid "..."))
)
```

Key rules:
* `lib_id` matches a symbol declared in `lib_symbols`.
* `at` positions are in millimetres. Rotation is 0, 90, 180, or 270.
* Each `pin` node has a stable UUID for net connectivity.
* Custom properties (e.g. `LCSC`, `MPN`) are added as extra
  `(property ...)` nodes after the standard four.
* `uuid` must be deterministic (see в”¬Вє 5).

### 1.4  Wires

Wires connect pin endpoints to form nets:

```sexpr
(wire
  (pts (xy 100.33 63.5) (xy 110.49 63.5))
  (stroke (width 0) (type default))
  (uuid "...")
)
```

Two pins are electrically connected when a chain of wires forms a
continuous path between their coordinates.  All wire segment endpoints
must land exactly on a pin endpoint or on another wire endpoint
(T-junction); floating endpoints cause ERC "unconnected wire" errors.

### 1.5  Labels, power ports, and global labels

| Element | Purpose | Example |
|---|---|---|
| `(label "I2C0_SDA" (at x y rot) ...)` | Local net label О“Г‡Г¶ names a net within the sheet | Signal nets |
| `(global_label "I2C0_SDA" (at x y rot) (shape input) ...)` | Cross-sheet net label | Multi-sheet designs |
| `(power_port ...)` | Power symbol instance (GND, +3V3) О“Г‡Г¶ behaves like a global label with a power flag | Power rails |

For the single-sheet MVP, `label` nodes are sufficient for signal nets.
Power nets (`GND`, `3V3`, `VBAT`, `SENSOR_3V3`, `VUSB`, `VIN`) are
represented by power symbol instances (e.g. `power:GND`,
`power:+3V3`) which implicitly create globally-scoped nets with the
power flag set (preventing ERC "power pin not driven" errors).

### 1.6  Reference documentation

* [KiCad 8 File Formats](https://dev-docs.kicad.org/en/file-formats/)
  О“Г‡Г¶ authoritative format specification.
* `kiutils` library source О“Г‡Г¶ practical examples of reading/writing
  every node type.

## 2  Template System

### 2.1  Concept

Each functional block of the schematic is a **template** О“Г‡Г¶ a Python
function that returns a list of KiCad S-expression elements (symbol
instances, wires, labels).  Templates are pure functions: given a
configuration fragment and a placement origin, they always produce
the same output.

```
Template function signature (conceptual):

def template_<name>(
    config: dict,              # relevant config fragment
    origin: tuple[float,float],# (x, y) placement origin in mm
    ref_alloc: RefAllocator,   # assigns next R, C, U, J, etc.
    uuid_gen: UuidGenerator,   # deterministic UUID factory
) -> TemplateResult
```

`TemplateResult` is a dataclass:

```python
@dataclass
class TemplateResult:
    lib_symbols: list[LibSymbol]   # symbol definitions needed
    instances: list[SymbolInstance] # placed components
    wires: list[Wire]              # wire segments
    labels: list[Label]            # net labels
    interface_nets: dict[str, tuple[float,float]]
        # net_name О“ГҐГ† pin coordinate exposed to the compositor
    properties: dict[str, str]     # metadata (e.g. contract entries)
```

### 2.2  Template catalog

| Template | Always included? | Contents |
|---|---|---|
| `base` | Yes | ESP32-C3 module (U1), MCP1700 LDO (U2), USBLC6-2SC6 ESD (U3), USB-C (J4), Schottky diodes (D1, D2), EN reset circuit (R6, C1, SW2), BOOT button (R5, SW1), strapping pull-ups (R3, R4), VDD bypass (C6), LDO decoupling (C3, C4, C5), USB series resistors (R1, R2), USB CC resistors (R13, R14) |
| `qwiic` | If `connectors.qwiic > 0` | Qwiic connector (J*n*), wired to `I2C0_SDA`, `I2C0_SCL`, `SENSOR_3V3`, `GND`.  I2C pull-ups (R7, R8) emitted only for the *first* Qwiic instance. |
| `battery` | If `connectors.battery` | JST-PH connector (J3), voltage divider (R11, R12), ADC filter cap (C2), wired to `VBAT`, `VBAT_SENSE`, `GND` |
| `power_gate` | If `power.sensor_gating` | Si2301 P-FET (Q1), gate pull-up (R9), gate drive resistor (R10), solder jumper (SJ1), wired to `3V3`, `SENSOR_3V3`, `SENSOR_PWR_EN` |
| `gpio_header` | If `connectors.gpio_header` | 2в”њГ№5 pin header (J6), wired per `gpio_header_pins` config |
| `spi_header` | If `connectors.spi` | SPI header (J5), wired to `SPI0_*` nets |
| `one_wire` | If `connectors.one_wire` | 1-Wire header + pull-up, wired to `OW0_DATA` |
| `sensor_<name>` | Per `sensors[]` list | Sensor IC + decoupling, wired to its bus |

### 2.3  Interface nets

Each template declares its **interface nets** О“Г‡Г¶ the named nets it
exposes for the compositor to connect to other templates.  Interface
nets are matched by name.

Example: the `qwiic` template exposes:

```python
interface_nets = {
    "I2C0_SDA":    (x1, y1),
    "I2C0_SCL":    (x2, y2),
    "SENSOR_3V3":  (x3, y3),
    "GND":         (x4, y4),
}
```

The `base` template exposes the same net names on the MCU pins.
The compositor connects them by generating wire segments or labels
that merge the endpoints.

### 2.4  Compositor algorithm

```
1. Instantiate `base` template at the origin.
2. For each enabled peripheral template (order: power_gate, battery,
   qwiicв”њГ№N, spi_header, one_wire, gpio_header, sensors):
   a. Call the template function with the next available placement
      origin (column-based layout: base left, peripherals right).
   b. Collect its TemplateResult.
3. Merge lib_symbols (deduplicate by symbol name).
4. Concatenate instances, wires, labels.
5. For each interface net that appears in multiple templates:
   - If templates share the same net name, place a `(label)` at
     each exposed coordinate.  KiCad connects all labels with
     identical names into one net.
6. Add power port symbols (GND, +3V3) at all power pin locations.
7. Assign reference designators (see в”¬Вє 2.5).
8. Serialize to `.kicad_sch`.
```

### 2.5  Reference designator assignment

Deterministic assignment per hw-design.md в”¬Вє 3.3:

```
Block order (fixed):
  base О“ГҐГ† power_gate О“ГҐГ† battery О“ГҐГ† qwiic[0] О“ГҐГ† qwiic[1] О“ГҐГ†
  spi_header О“ГҐГ† one_wire О“ГҐГ† gpio_header О“ГҐГ† sensors[0..N]

Within each block, components are ordered by type prefix (C, D, J,
Q, R, SJ, SW, U) then by the template's internal declaration order.

Counters:
  U: 1   (U1=MCU, U2=LDO, U3=ESD, U10+=sensors)
  J: 1   (per design table)
  Q: 1
  D: 1
  R: 1   (global counter across all blocks)
  C: 1   (global counter across all blocks)
  SW: 1
  SJ: 1
```

Each template does **not** hard-code reference designators.  Instead,
it uses placeholder labels (e.g. `"MCU"`, `"LDO"`) and the
`RefAllocator` assigns final designators after all templates are
collected, in the fixed block order above.

## 3  Component Library Strategy

### 3.1  KiCad standard library symbols

The following parts use symbol shapes defined in the KiCad 8 standard
library (`kicad-symbols`).  For HW-0900 determinism, the tool does
**not** read these from the host KiCad installation at generation time.
Instead, each required stdlib symbol is vendored (copied) into the
pinned `hw/symbols/sonde.kicad_sym` library and sourced exclusively
from there (see в”¬Вє3.2).

| Component | KiCad Library Symbol |
|---|---|
| Resistor | `Device:R` |
| Capacitor | `Device:C` |
| Schottky diode | `Device:D_Schottky` |
| P-FET MOSFET | `Device:Q_PMOS_GSD` |
| Tactile switch | `Switch:SW_Push` |
| Solder jumper NO | `Jumper:SolderJumper_2_Open` |
| USB ESD (USBLC6-2SC6) | `Power_Protection:USBLC6-2SC6` |
| Generic pin header | `Connector_Generic:Conn_02x05_Odd_Even` |
| Power symbols | `power:GND`, `power:+3V3`, `power:+5V` |

### 3.2  Custom symbols

Parts not in the KiCad standard library require custom symbols stored
in `hw/symbols/sonde.kicad_sym`:

| Component | Custom Symbol Name | Reason |
|---|---|---|
| ESP32-C3-MINI-1 | `sonde:ESP32-C3-MINI-1` | Module pinout not in stdlib (verify each KiCad release) |
| Qwiic connector (JST-SH 4P) | `sonde:Qwiic_Connector` | Specific pin labeling (GND, VCC, SDA, SCL) |
| JST-PH 2P battery | `Connector_JST:JST_PH_S2B-PH-SM4-TB_1x02-1MP_Horizontal` | Stdlib symbol shape; definition is pinned (vendored) in `hw/symbols/sonde.kicad_sym` |
| USB-C 16-pin | `Connector_USB:USB_C_Receptacle_USB2.0` | Stdlib symbol shape; definition is pinned (vendored) in `hw/symbols/sonde.kicad_sym` |

The tool **must always** source symbol definitions from the pinned
`hw/symbols/sonde.kicad_sym` library.  Generated `lib_symbols` content
must be independent of the host KiCad installation to satisfy HW-0900
determinism.  If any required symbol is missing from the pinned library,
the tool fails with a hard error rather than falling back to stdlib or
any environment-dependent source.

### 3.3  Footprint library paths

```
Footprints are resolved from:
1. KiCad stdlib:  ${KICAD8_FOOTPRINT_DIR}/*.pretty/
2. Custom:        hw/footprints/*.pretty/
```

Custom footprints are needed only for parts absent from the KiCad
footprint libraries (e.g. a specific JST-SH 4-pin horizontal SMD
variant).  All standard passives (0402 R, C), SOT-23, SMA, etc. are
available in the stdlib.

### 3.4  LCSC part number annotation

Every component instance carries an `LCSC` property:

```sexpr
(property "LCSC" "C25744" (at x y 0)
  (effects (font (size 1.27 1.27)) hide))
```

This property is:
* Populated from `bom_overrides` in the YAML config if present.
* Populated from a built-in part database (`sonde_hw/parts_db.py`)
  as a fallback.
* Exported to the BOM CSV for JLCPCB assembly ordering.

## 4  Generation Pipeline

The full pipeline (`sonde-hw build <config.yaml>`) executes these
steps in order.  Each step is also available as a standalone command.

### 4.1  Step 1 О“Г‡Г¶ `validate`

**Command:** `sonde-hw validate <config.yaml>`

**Inputs:** YAML config file, JSON Schema (`hw/schema.json`).

**Process:**

1. Load YAML with PyYAML using a custom loader that rejects duplicate
   keys (for example, a subclass of `yaml.SafeLoader` that errors on
   duplicate mapping keys).
2. Validate the resulting document against `hw/schema.json` using
   `jsonschema.validate()`.  The schema enforces:
   - Required keys: `mcu`, `flash_size`, `connectors`, `power`,
     `board`, `pins`.
   - Type constraints: `connectors.qwiic` is integer 0О“Г‡Гґ2,
     `power.sensor_gating` is boolean, etc.
   - `additionalProperties: false` at every object level (strict
     parsing per HW-0700 AC-3).
3. Run semantic checks (not expressible in JSON Schema):
   - **I2C address conflicts:** no two entries in `sensors[]` share
     the same I2C address.
   - **Pin conflicts:** no GPIO number appears in more than one role
     across `pins`, `power.sensor_gate_gpio`, `power.battery_adc_gpio`,
     and `gpio_header_pins[].net`.
   - **Size feasibility:** estimated component count fits the
     selected `board.size` variant (compact О“Г«Г± 25 components,
     standard О“Г«Г± 50 components).
   - **Power budget estimate:** sum of always-on loads (regulator Iq +
     pull-up leakage + divider leakage) О“Г«Г± 20 в”¬в•ЎA (HW-0400).

**Outputs:** exit 0 on success; structured error list on failure.

### 4.2  Step 2 О“Г‡Г¶ `generate`

**Command:** implicit in `sonde-hw build`

**Inputs:** validated config, template functions, parts database.

**Process:**

1. Compute `config_hash = SHA-256(canonical YAML text)`.
2. Initialize `UuidGenerator(config_hash)` (в”¬Вє 5.1).
3. Initialize `RefAllocator()`.
4. Call template functions in fixed order (в”¬Вє 2.4) to produce
   `TemplateResult` objects.
5. Run the compositor:
   a. Merge `lib_symbols` (deduplicate).
   b. Concatenate instances, wires, labels.
   c. Connect interface nets via labels.
   d. Insert power port symbols.
   e. Assign reference designators.
6. Build the title block with config hash, date, and tool version.
7. Serialize to `hw/output/<config-name>/board.kicad_sch` using
   `kiutils` S-expression writer.
8. Generate `hw/output/<config-name>/bom.csv` from the component
   list (reference, value, footprint, MPN, LCSC, quantity).
9. Generate `hw/output/<config-name>/contract.yaml` (в”¬Вє 4.7).

**Outputs:** `board.kicad_sch`, `bom.csv`, `contract.yaml`.

### 4.3  Step 3 О“Г‡Г¶ `erc`

**Command:** implicit in `sonde-hw build`

**Inputs:** generated `board.kicad_sch`.

**Process:**

```
kicad-cli sch erc board.kicad_sch \
    --exit-code-violations \
    --output erc-report.json \
    --format json
```

The tool parses the JSON report.  If violations exist, it prints
them and exits non-zero.  Expected warnings (e.g. intentionally
unconnected pins on the ESP32-C3 module) are suppressed by listing
them in a `hw/erc-exclusions.json` file.

**Outputs:** `erc-report.json` (kept for CI artifact upload).

### 4.4  Step 4 О“Г‡Г¶ `place` (post-MVP)

**Command:** implicit in `sonde-hw build` (skipped in MVP).

**Inputs:** `board.kicad_sch`, board outline parameters.

**Process:**

1. Create a blank `.kicad_pcb` with:
   - Layer stackup (F.Cu, B.Cu, F.SilkS, B.SilkS, F.Mask, B.Mask,
     Edge.Cuts).
   - Board outline on Edge.Cuts (rectangle per `board.size`).
   - Mounting hole footprints at fixed positions.
   - Antenna keepout zone on F.Cu and B.Cu.
2. Import the netlist from the schematic (via `kicad-cli` or
   `kiutils` netlist export).
3. Place components deterministically per hw-design.md в”¬Вє 4.2:
   - USB-C at bottom edge center.
   - ESP32-C3 at board center, antenna toward top.
   - LDO near USB-C.
   - Decoupling caps within 3 mm of their IC.
   - Connectors along left/right edges.
4. Write `hw/output/<config-name>/board.kicad_pcb`.

**Outputs:** `board.kicad_pcb`.

> **MVP note:** this step is skipped.  Operators open the generated
> `.kicad_sch` in KiCad and perform manual layout.

### 4.5  Step 5 О“Г‡Г¶ `drc` (post-MVP)

**Command:** implicit in `sonde-hw build` (skipped in MVP).

**Inputs:** `board.kicad_pcb`, fab DRC rules.

**Process:**

```
kicad-cli pcb drc board.kicad_pcb \
    --exit-code-violations \
    --output drc-report.json \
    --format json \
    --schematic board.kicad_sch \
    --severity-all
```

Loads fab-specific rules from `hw/rules/<fab>.kicad_dru` (e.g.
`jlcpcb.kicad_dru`).

**Outputs:** `drc-report.json`.

### 4.6  Step 6 О“Г‡Г¶ `export` (post-MVP)

**Command:** `sonde-hw export <config.yaml>`

**Inputs:** `board.kicad_pcb`.

**Process:**

```bash
# Gerber export
kicad-cli pcb export gerbers board.kicad_pcb \
    --output gerber/ \
    --layers "F.Cu,B.Cu,F.SilkS,B.SilkS,F.Mask,B.Mask,Edge.Cuts"

# Drill file
kicad-cli pcb export drill board.kicad_pcb \
    --output gerber/ \
    --format excellon

# Pick-and-place / CPL
kicad-cli pcb export pos board.kicad_pcb \
    --output cpl.csv \
    --format csv \
    --units mm
```

**Outputs:** `gerber/` directory, `bom.csv`, `cpl.csv`.

### 4.7  Step 7 О“Г‡Г¶ `contract`

**Command:** implicit in `sonde-hw build` (runs during `generate`).

**Inputs:** config, template metadata, ESP32-C3 datasheet constants.

**Process:**

Each template contributes contract entries via its `properties` dict.
The compositor merges them into a single `contract.yaml`:

```yaml
contract_version: 1
config_hash: "<sha256>"
board: "<config-name>"

rails:
  - name: "3V3"
    type: regulated
    voltage: { min: 3.2, typ: 3.3, max: 3.4 }
    source: "MCP1700-3302E/TT"
    budget:
      deep_sleep_ua: 7.0
      active_ma: 120
      radio_tx_burst_ma: 340
  - name: "SENSOR_3V3"
    type: switched
    voltage: { min: 3.1, typ: 3.3, max: 3.4 }
    source: "Si2301 P-FET from 3V3"
    enable_pin: "GPIO3"
    enable_active: low
  # ...

pins:
  - mcu_pin: "GPIO4"
    net: "I2C0_SDA"
    role: bidirectional
    peripheral: "I2C0"
    pull: { type: pull_up, value_ohm: 4700, rail: "SENSOR_3V3" }
    # ...
  # ...

operating_states:
  - name: deep_sleep
    rails: { "3V3": on, "SENSOR_3V3": off }
  - name: active
    rails: { "3V3": on, "SENSOR_3V3": on }
  # ...
```

The contract is validated against `hw/contract-schema.json` before
being written.

## 5  Determinism (HW-0900)

### 5.1  UUID generation

KiCad requires a UUID for every element (components, pins, wires,
labels).  The tool generates deterministic UUIDs:

```python
import hashlib, uuid

class UuidGenerator:
    def __init__(self, config_hash: str):
        self._seed = bytes.fromhex(config_hash)

    def next(self, path: str) -> str:
        """Generate a deterministic UUID from config hash + path.

        `path` is a unique key like "U1", "U1:pin:1", "wire:3".
        """
        h = hashlib.sha256(self._seed + path.encode()).digest()
        return str(uuid.UUID(bytes=h[:16], version=4))
```

The `path` argument is constructed from the component's block name,
type prefix, and sequence index (e.g. `"base/U/0"`, `"qwiic_0/J/0"`,
`"base/wire/12"`).  This ensures UUIDs are stable across runs and
change only when the config changes.

### 5.2  Timestamp handling

* **Title block date:** set to a fixed, deterministic constant
  `date = "2026-01-01"` (fixed epoch) for all generated designs.
  This avoids reliance on system time and keeps outputs stable
  across runs for the same input config.
* **File modification time:** not embedded in KiCad files.  The tool
  does not call any API that inserts `os.time()` or `datetime.now()`.
* **`generator_version`:** hardcoded string from the tool's version
  constant.

### 5.3  Sort order

All lists that could vary in order are sorted before serialization:

| Collection | Sort key |
|---|---|
| `lib_symbols` | Library-qualified name (`"Device:C"` < `"Device:R"`) |
| Component instances | Reference designator (`C1` < `C2` < `R1` < `U1`) |
| Wires | `(min(x1,x2), min(y1,y2), max(x1,x2), max(y1,y2))` |
| Labels | `(name, x, y)` |
| BOM rows | Reference designator |
| Contract rails | Rail name |
| Contract pins | MCU pin name (numeric sort on GPIO number) |

### 5.4  Verification

To verify bit-identical output:

```bash
# Generate twice, diff
sonde-hw build hw/configs/minimal-qwiic.yaml
cp -r hw/output/minimal-qwiic hw/output/minimal-qwiic-run1

sonde-hw build hw/configs/minimal-qwiic.yaml
diff -r hw/output/minimal-qwiic hw/output/minimal-qwiic-run1
# Expected: no differences
```

CI runs this as a determinism check (see в”¬Вє 8.3).

## 7  Testing Strategy

### 7.1  Unit tests

| Test file | What it covers |
|---|---|
| `test_config.py` | YAML loading, schema validation, semantic checks (pin conflicts, address conflicts, power budget) |
| `test_templates.py` | Each template function in isolation: correct component count, correct interface nets, correct wire connectivity |
| `test_compositor.py` | Net merging across templates, label deduplication, power port insertion |
| `test_ref_alloc.py` | Reference designator assignment order, counter continuity across blocks |
| `test_uuid_gen.py` | Deterministic output, uniqueness, stability across calls |
| `test_serializer.py` | S-expression output format correctness, round-trip with `kiutils` parser |
| `test_bom.py` | BOM CSV format, LCSC column, quantity rollup |

### 7.2  Integration tests

| Test | What it verifies |
|---|---|
| `test_erc_integration.py` | Generate `minimal-qwiic` schematic О“ГҐГ† run `kicad-cli sch erc` О“ГҐГ† assert zero violations. **Requires KiCad 8 installed.** Marked with `@pytest.mark.integration` and skipped if `kicad-cli` is not on `PATH`. |
| `test_determinism.py` | Generate the same config twice О“ГҐГ† assert byte-identical `.kicad_sch` output. |

### 7.3  Snapshot (golden file) tests

The `golden/` directory contains reference output files.  Tests
generate output and compare against the golden file:

```python
def test_minimal_qwiic_snapshot(snapshot):
    result = generate("hw/configs/minimal-qwiic.yaml")
    snapshot.assert_match(result, "minimal-qwiic.kicad_sch")
```

Golden files are updated with `pytest --snapshot-update` when
intentional changes are made to templates or the serializer.

## 8  MVP Scope

The minimum viable first version targets **schematic generation and
ERC** for the `minimal-qwiic` board configuration only.

### 8.1  MVP includes

| Feature | Details |
|---|---|
| `sonde-hw validate` | JSON Schema + semantic validation of YAML config |
| `sonde-hw build` (schematic only) | Generate `board.kicad_sch` for `minimal-qwiic` |
| ERC | Run `kicad-cli sch erc`, report results |
| BOM export | Generate `bom.csv` from schematic component list |
| Contract generation | Generate `contract.yaml` |
| Determinism | Bit-identical output from identical input |
| Templates | `base`, `qwiic`, `battery`, `power_gate`, `gpio_header` |
| Unit tests | Full coverage of config, templates, compositor, serializer |
| Integration test | Generate + ERC pass for `minimal-qwiic` |
| Snapshot test | Golden file for `minimal-qwiic.kicad_sch` |

### 8.2  MVP excludes (deferred to later phases)

| Feature | Reason |
|---|---|
| PCB layout generation (`.kicad_pcb`) | Manual layout in KiCad for v1; autoplace is complex |
| DRC | Requires `.kicad_pcb` |
| Gerber / CPL export | Requires `.kicad_pcb` |
| SPI header template | Not needed for `minimal-qwiic` |
| 1-Wire template | Not needed for `minimal-qwiic` |
| Sensor templates | Not needed for `minimal-qwiic` (no on-board sensors) |
| `sonde-hw budget` command | Power budget calc is a nice-to-have |
| `sonde-hw check-firmware` command | Requires contract + firmware NVS config integration |
| Multi-sheet schematics | Single sheet is sufficient for О“Г«Г± 50 components |

### 8.3  MVP acceptance criteria

1. `sonde-hw validate hw/configs/minimal-qwiic.yaml` exits 0.
2. `sonde-hw build hw/configs/minimal-qwiic.yaml` produces
   `hw/output/minimal-qwiic/board.kicad_sch` that opens correctly
   in KiCad 8.
3. `kicad-cli sch erc hw/output/minimal-qwiic/board.kicad_sch
   --exit-code-violations` exits 0 (zero ERC errors).
4. `hw/output/minimal-qwiic/bom.csv` lists all 33 components from
   the schematic design doc (в”¬Вє 9 of `hw-schematic-design.md`).
5. Running `sonde-hw build` twice produces byte-identical output.
6. `pytest hw/tests/` passes with 100% of tests green.
