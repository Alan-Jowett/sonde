## Issue 2: Ambiguous byte range notation in key_hint derivation
**File:** docs/ble-pairing-protocol.md:71,82
**Severity:** Medium
**Problem:** The notation `SHA256(psk)[30..32]` for key_hint derivation is ambiguous and inconsistent with typical array indexing conventions.

From §2.2 and §2.3:
  phone_key_hint = SHA256(phone_psk)[30..32]
  node_key_hint = SHA256(node_psk)[30..32]

The notation [30..32] could be interpreted as:
1. Python-style range [30:32] = bytes at indices 30 and 31 (correct interpretation)
2. Inclusive range [30,31,32] = 3 bytes (incorrect)
3. Mathematical notation [30..32] = ambiguous

§11 clarifies this means "Lower 16 bits" (last 2 bytes), but implementers might misread [30..32] as 3 bytes.

**Evidence:** SHA-256 produces 32 bytes (indices 0-31). The last 2 bytes are at indices 30 and 31. The notation should unambiguously specify 2 bytes, not leave room for interpretation as indices 30, 31, and 32.

**Impact:** Could cause implementers to extract wrong bytes or extract 3 bytes instead of 2.

**Suggested fix:** Use explicit notation:
  - "SHA256(psk)[30:32]" (Python slice notation)
  - "The last 2 bytes of SHA256(psk)"
  - "SHA256(psk) bytes 30-31 (zero-indexed)"

## Issue 5: PEER_ACK registration_proof timing attack surface
**File:** docs/ble-pairing-protocol.md:498,503
**Severity:** Low (security consideration, not a bug)
**Problem:** The registration_proof verification (step 3-4 in §8.3) compares SHA-256 hashes using standard equality, which could be vulnerable to timing attacks if the comparison is not constant-time.

From §8.3:
  "3. Compute expected `registration_proof` = `SHA256(node_psk ‖ encrypted_payload)` using the stored encrypted payload from NVS.
   4. If `registration_proof` does not match → discard (potential forgery)."

**Evidence:** No mention of constant-time comparison for registration_proof.
**Impact:** A sophisticated attacker with precise timing measurement could potentially extract information about the expected registration_proof through timing side-channels. However, this is a low-severity issue because:
  - The attacker would need to control the radio channel to send forged PEER_ACK messages
  - The registration_proof is already protected by frame HMAC verification
  - The attack surface is limited to a brief window during initial node registration

**Suggested fix:** Add a note in §8.3 or §9 that registration_proof comparison SHOULD use constant-time comparison to prevent timing side-channels, or document why this is acceptable risk for this protocol.
