# BLE Pairing Tool Requirements Specification

> **Document status:** Draft  
> **Source:** Derived from [ble-pairing-protocol.md](ble-pairing-protocol.md) (BLE pairing protocol specification).  
> **Scope:** This document covers the Sonde **BLE pairing tool** (phone/operator-side) only. Gateway-side BLE GATT services and node firmware BLE behavior are out of scope.  
> **Related:** [ble-pairing-protocol.md](ble-pairing-protocol.md), [security.md](security.md), [gateway-requirements.md](gateway-requirements.md), [node-requirements.md](node-requirements.md)

---

## 1  Definitions

| Term | Definition |
|---|---|
| **Phase 1** | Phone-to-gateway pairing: phone authenticates gateway, receives a unique phone PSK. One-time per gateway. |
| **Phase 2** | Node provisioning: phone generates a node PSK, provisions the node over BLE, and constructs an encrypted payload for the gateway. Per-node. |
| **Phase 3** | Node registration: node transmits `PEER_REQUEST` to gateway over ESP-NOW. Firmware responsibility — not part of the tool. |
| **TOFU** | Trust-on-first-use: phone pins the gateway public key on first contact and rejects mismatches thereafter. |
| **Phone PSK** | 256-bit pre-shared key issued by the gateway to the phone during Phase 1. Authenticates all subsequent pairing requests. |
| **Node PSK** | 256-bit pre-shared key generated by the phone for each node during Phase 2. Used for HMAC-SHA256 authentication on ESP-NOW. |
| **key_hint** | `u16::from_be_bytes(SHA-256(psk)[30..32])` — a 16-bit non-unique lookup hint derived from a PSK. |
| **gateway_id** | 16-byte random identifier generated on first gateway startup. Used as HKDF salt and AES-256-GCM AAD. |
| **encrypted_payload** | Opaque blob constructed by the phone (Phase 2), stored by the node, relayed to the gateway via ESP-NOW `PEER_REQUEST`. |
| **Registration window** | Time-limited period (default 120 s) during which the gateway accepts `REGISTER_PHONE` requests. Opened by operator action. |

## 2  Requirement format

Each requirement uses the following fields:

- **ID** — Unique identifier (`PT-XXXX`).
- **Title** — Short name.
- **Description** — What the pairing tool must do.
- **Acceptance criteria** — Observable, testable conditions that confirm the requirement is met.
- **Priority** — MoSCoW: **Must**, **Should**, **May**.
- **Source** — Document and section that motivates the requirement.

## 3  Platform and architecture

### PT-0100  Supported platforms

**Priority:** Must  
**Source:** Product requirements §2.1

**Description:**  
The initial release MUST support **Windows** (desktop, native Bluetooth stack) and **Android** (physical devices, Android BLE API). The design MUST NOT preclude adding iOS later.

**Acceptance criteria:**

1. The tool builds and runs on Windows (`x86_64-pc-windows-msvc`).
2. The tool builds and runs on Android (ARM64 via NDK) on physical hardware.
3. No iOS-only APIs are used; no assumptions prevent a future iOS port.

---

### PT-0101  Rust-first implementation

**Priority:** Must  
**Source:** Product requirements §3.1

**Description:**  
Core logic (pairing state machine, cryptography, validation, persistence) MUST be implemented in Rust. The UI layer is a thin shell over Rust commands, following a Tauri-style architecture (Rust backend + minimal frontend).

**Acceptance criteria:**

1. All protocol logic, message construction/parsing, and cryptographic operations are in Rust.
2. The UI layer invokes Rust functions and does not contain protocol logic.

---

### PT-0102  Platform isolation

**Priority:** Must  
**Source:** Product requirements §3.2

**Description:**  
All platform-specific BLE code MUST be isolated behind a `BleTransport` trait (or equivalent abstraction). All platform-specific storage code MUST be behind a `PairingStore` trait. The core pairing crate MUST compile on any `std` Rust target without platform dependencies.

**Acceptance criteria:**

1. The core `sonde-pair` crate builds on `x86_64-pc-windows-msvc`, `aarch64-linux-android`, and `x86_64-unknown-linux-gnu` without modification.
2. Adding a new platform requires only a new trait implementation — no changes to core logic.
3. Tests run against mock implementations of both traits.

---

### PT-0103  Crate placement

**Priority:** Should  
**Source:** Product requirements §3.1

**Description:**  
The pairing logic SHOULD live in a new workspace crate (e.g., `sonde-pair`) that depends on `sonde-protocol` for CBOR, HMAC, and SHA-256 utilities and does NOT depend on `sonde-gateway`, `sonde-node`, or `sonde-modem`.

**Acceptance criteria:**

1. The crate is a member of the Cargo workspace.
2. The crate's `Cargo.toml` does not list `sonde-gateway`, `sonde-node`, or `sonde-modem` as dependencies.
3. The pairing state machine is reusable by future CLIs, tests, or alternative frontends.

---

### PT-0104  Separation of concerns

**Priority:** Must  
**Source:** Product requirements §3.2

**Description:**  
The design MUST clearly separate four layers: **protocol logic** (state machines, message construction, crypto), **transport** (BLE scanning, GATT operations), **persistence** (storing/loading pairing artifacts), and **UI** (scan/select/pair/status display).

**Acceptance criteria:**

1. Each layer is implemented in a distinct Rust module or set of modules.
2. No BLE platform calls appear in protocol logic modules.
3. No UI code appears in protocol or transport modules.

---

### PT-0105  Android BLE runtime permissions

**Priority:** Must  
**Source:** ble-pairing-tool-design.md §9.2

**Description:**  
On Android, BLE scanning and connection require **runtime permissions** that the OS does not grant automatically at install time.  The app MUST request `BLUETOOTH_SCAN` and `BLUETOOTH_CONNECT` (API 31+) or `ACCESS_FINE_LOCATION` (API 23–30) before the first BLE operation.  The user MUST see a system consent dialog; the app MUST NOT silently fail if permissions are not yet granted.

**Acceptance criteria:**

1. On first launch with no prior grants, the Android system permission dialog appears before any BLE operation is attempted.
2. If the user grants the permissions, scanning and connection proceed normally.
3. If the user denies the permissions, the app surfaces an actionable error message (not a silent failure or crash).
4. On API 31+ the requested permissions are `BLUETOOTH_SCAN` and `BLUETOOTH_CONNECT`; on API 23–30 the requested permission is `ACCESS_FINE_LOCATION`.

---

### PT-0106  LESC Numeric Comparison pairing

**Priority:** Must  
**Source:** ble-pairing-tool-design.md §9.1, §9.2; security.md

**Description:**  
BLE connections to the modem MUST use **LE Secure Connections (LESC) with Numeric Comparison** as the pairing method.  This ensures mutual authentication and MITM protection during the BLE link establishment.  The app MUST NOT fall back to **Just Works** pairing silently; if the platform cannot complete Numeric Comparison (e.g., the modem does not support `DisplayYesNo` I/O capability), the connection MUST fail with an actionable error.

**Acceptance criteria:**

1. **Precondition:** the modem is configured for LESC with `DisplayYesNo` I/O capability (see modem requirements).
2. The phone app triggers `createBond()` (Android) or equivalent platform API to initiate LESC pairing, **or** responds to a server-initiated SMP Security Request from the modem (MD-0404 criterion 5). On platforms where the BLE stack handles server-initiated pairing transparently (e.g. WinRT, CoreBluetooth), the app need not call a pairing API explicitly.
3. A Numeric Comparison dialog is presented to the user for confirmation.
4. If the user rejects the comparison or the pairing mode degrades to Just Works, the connection is terminated.

---

### PT-0107  Android activity lifecycle management

**Priority:** Should  
**Source:** ble-pairing-tool-design.md §9.2

**Description:**  
On Android, the BLE transport SHOULD disconnect cleanly when the hosting Activity is paused (e.g., user switches apps) and reconnect on resume if a pairing flow was in progress.  This prevents leaked BLE connections and stale GATT state.

**Acceptance criteria:**

1. Backgrounding the app mid-pairing disconnects the BLE link within 5 s.
2. Returning to the app after a lifecycle-triggered disconnect surfaces an error or resumes the flow cleanly (no crash, no hang).
3. No GATT connections remain open after the Activity is destroyed.

---

### PT-0108  JNI classloader caching

**Priority:** Must  
**Source:** ble-pairing-tool-design.md §9.2 (implicit); Android JNI best practices

**Description:**  
On Android, the native library MUST cache `GlobalRef` references to app-defined Java classes (e.g., `BleHelper`, `SecureStore`) during `JNI_OnLoad` (or another Java-attached thread with the application classloader).  Threads attached via `JavaVM::attach_current_thread()` (e.g., tokio worker threads) use the system classloader, which cannot find app-defined classes via `FindClass`.

**Acceptance criteria:**

1. `JNI_OnLoad` resolves and caches `GlobalRef` for every app-defined class used from Rust.
2. BLE and storage operations succeed when invoked from tokio background threads.
3. No `ClassNotFoundException` or `FindClass` failure occurs on non-main threads.

## 4  Device discovery

### PT-0200  BLE scanning

**Priority:** Must  
**Source:** ble-pairing-protocol.md §3.1

**Description:**  
The tool MUST scan for BLE peripherals advertising either the **Gateway Pairing Service** UUID (`0000FE60-0000-1000-8000-00805F9B34FB`) or the **Node Provisioning Service** UUID (`0000FE50-0000-1000-8000-00805F9B34FB`).

**Acceptance criteria:**

1. Devices advertising the Gateway Pairing Service UUID appear in scan results.
2. Devices advertising the Node Provisioning Service UUID appear in scan results.
3. Devices not advertising either UUID do not appear.

---

### PT-0201  Device presentation

**Priority:** Must  
**Source:** Product requirements §4.1

**Description:**  
Each discovered device MUST be presented with: BLE advertising name (e.g., `"sonde-XXXX"` for nodes), service type (gateway vs. node), and signal strength (RSSI) if available from the platform.

**Acceptance criteria:**

1. The device list shows the advertising name, service type, and RSSI for each device.
2. Gateways and nodes are visually distinguishable (e.g., by label or icon).

---

### PT-0202  Scan lifecycle

**Priority:** Must  
**Source:** Product requirements §4.1

**Description:**  
The operator MUST be able to start and stop scanning. Scanning MUST time out after a configurable period (default: 30 s). Devices that stop advertising for > 10 s MUST be removed from the list.

**Acceptance criteria:**

1. The operator can start and stop scanning via the UI.
2. Scanning stops automatically after the configured timeout.
3. Stale devices are removed from the list within 10 s of ceasing to advertise.

## 5  Phase 1 — Gateway pairing

### PT-0300  BLE connection and MTU

**Priority:** Must  
**Source:** ble-pairing-protocol.md §3.4

**Description:**  
On operator selection of a gateway device, the tool MUST connect to the BLE peripheral, negotiate ATT MTU ≥ 247, and accept BLE LESC pairing.  Numeric Comparison is the required method for the gateway pairing service (the tool displays or relays the 6-digit passkey for operator verification).  Just Works MUST NOT be accepted because it provides no MITM protection and the pairing exchange carries PSK material (see PT-0106).  If the negotiated MTU is < 247, the tool MUST disconnect and report an error.

**Acceptance criteria:**

1. The tool connects to the selected gateway.
2. MTU ≥ 247 is negotiated.
3. If MTU < 247, the tool disconnects and reports "MTU too low — device requires MTU ≥ 247".

---

### PT-0301  Gateway authentication (REQUEST_GW_INFO / GW_INFO_RESPONSE)

**Priority:** Must  
**Source:** ble-pairing-protocol.md §5.1, §5.2

**Description:**  
The tool MUST generate a 32-byte challenge from OS CSPRNG, write `REQUEST_GW_INFO` to the Gateway Command characteristic, then wait for `GW_INFO_RESPONSE` (timeout: 45 s from completion of the write) and verify the Ed25519 signature over `(challenge ‖ gateway_id)` using the received `gw_public_key`. On failure, the tool MUST disconnect and report "gateway authentication failed — possible impersonation". The 45 s timeout applies only to waiting for the `GW_INFO_RESPONSE` indication after `REQUEST_GW_INFO` has been written. On Windows, the underlying WinRT GATT write of `REQUEST_GW_INFO` may itself be internally retried for up to ~30 s in response to authentication errors (for example, while the operator confirms the Numeric Comparison passkey), so the total wall-clock time from write initiation to timeout can be up to ~75 s.

**Acceptance criteria:**

1. A 32-byte challenge is generated from OS CSPRNG for each attempt.
2. The tool writes `REQUEST_GW_INFO` and waits for the indication.
3. On timeout (45 s) the tool disconnects and reports a timeout error.
4. On signature verification failure the tool disconnects and reports authentication failure.
5. On success the tool proceeds to phone registration.

---

### PT-0302  Trust-on-first-use (TOFU)

**Priority:** Must  
**Source:** ble-pairing-protocol.md §5.1

**Description:**  
On first successful gateway authentication, the tool MUST persist `gw_public_key` and `gateway_id` in the local pairing store. On subsequent connections, the tool MUST reject any gateway presenting a different public key and report the mismatch. The tool SHOULD offer a mechanism to clear the pinned gateway identity (explicit operator action).

**Acceptance criteria:**

1. After first successful Phase 1, `gw_public_key` and `gateway_id` are persisted.
2. A subsequent connection to a gateway with a different public key is rejected with an error.
3. An explicit operator action can clear the pinned identity.

---

### PT-0303  Phone registration (REGISTER_PHONE / PHONE_REGISTERED)

**Priority:** Must  
**Source:** ble-pairing-protocol.md §5.3, §5.4, §5.5

**Description:**  
The tool MUST generate an ephemeral X25519 keypair, write `REGISTER_PHONE` (ephemeral public key + operator-supplied label, max 64 bytes UTF-8) to the Gateway Command characteristic, and wait for a response (timeout: 30 s). On receiving `PHONE_REGISTERED`, the tool MUST decrypt it using ECDH + HKDF-SHA256 + AES-256-GCM (per protocol §5.5) and extract `phone_psk`, `phone_key_hint`, and `rf_channel`. On success, all artifacts MUST be persisted.

**Acceptance criteria:**

1. An ephemeral X25519 keypair is generated per registration attempt.
2. The tool writes `REGISTER_PHONE` and waits up to 30 s.
3. `ERROR(0x02)` → "registration window not open — ask operator to enable pairing".
4. `ERROR(0x03)` → "already paired with this gateway".
5. Timeout → "gateway did not respond — ensure registration window is open".
6. On successful decryption, `phone_psk`, `phone_key_hint`, `rf_channel`, `gw_public_key`, and `gateway_id` are persisted.
7. On decryption failure (bad GCM tag), the tool disconnects and reports "ephemeral key mismatch".

---

### PT-0304  Ephemeral key zeroing

**Priority:** Must  
**Source:** security.md §2.1

**Description:**  
The ephemeral X25519 private key MUST be wrapped in `zeroize::Zeroizing` and zeroed immediately after the ECDH shared secret is derived. The shared secret and derived AES key MUST also be zeroed after use.

**Acceptance criteria:**

1. Ephemeral private key, shared secret, and AES key are wrapped in `Zeroizing`.
2. After the ECDH derivation and decryption are complete, no copies of these values remain in memory.

---

## 6  Phase 2 — Node provisioning

### PT-0400  Phase 1 prerequisite check

**Priority:** Must  
**Source:** ble-pairing-protocol.md §6

**Description:**  
Before starting node provisioning, the tool MUST verify that it holds a valid phone PSK from a prior Phase 1 exchange. If not, it MUST prompt the operator to pair with a gateway first.

**Acceptance criteria:**

1. If no `phone_psk` is in the pairing store, the tool refuses to proceed and reports "no gateway pairing found — complete Phase 1 first".
2. If a valid `phone_psk` is present, the tool proceeds.

---

### PT-0401  Node BLE connection and MTU

**Priority:** Must  
**Source:** ble-pairing-protocol.md §3.4

**Description:**  
On operator selection of a node device, the tool MUST connect, negotiate ATT MTU ≥ 247, and accept BLE LESC pairing. If MTU < 247, the tool MUST disconnect and report an error.

**Acceptance criteria:**

1. The tool connects to the selected node.
2. MTU ≥ 247 is negotiated.
3. If MTU < 247, the tool disconnects and reports "MTU too low".

---

### PT-0402  Node PSK generation

**Priority:** Must  
**Source:** ble-pairing-protocol.md §6.1, security.md §2.1

**Description:**  
The tool MUST generate a 32-byte node PSK from OS CSPRNG (`getrandom::fill()`) and derive `node_key_hint` as `u16::from_be_bytes(SHA-256(node_psk)[30..32])`.

**Acceptance criteria:**

1. The node PSK is 32 bytes from OS CSPRNG.
2. `node_key_hint` equals the big-endian u16 from bytes 30–31 of `SHA-256(node_psk)`.

---

### PT-0403  PairingRequest CBOR construction

**Priority:** Must  
**Source:** ble-pairing-protocol.md §6.1

**Description:**  
The tool MUST build a PairingRequest as a deterministic CBOR map (RFC 8949 §4.2) with integer keys in ascending order, minimal-length integer encoding, and definite-length containers. Required fields: `node_id` (1–64 bytes UTF-8, operator-supplied), `node_key_hint`, `node_psk`, `rf_channel` (from Phase 1), `sensors` (optional), `timestamp` (current Unix time in seconds).

**Acceptance criteria:**

1. The CBOR output is deterministic — encoding the same fields twice produces identical bytes.
2. Integer keys are in ascending order (1, 2, 3, 4, 5, 6).
3. `node_id` is validated: empty strings and strings > 64 bytes are rejected.
4. `rf_channel` is validated: values outside 1–13 are rejected.

---

### PT-0404  Phone HMAC authentication

**Priority:** Must  
**Source:** ble-pairing-protocol.md §6.2

**Description:**  
The tool MUST authenticate the PairingRequest CBOR bytes with `HMAC-SHA256(phone_psk, cbor_bytes)`, then assemble the authenticated request as `phone_key_hint[2] ‖ cbor_bytes ‖ phone_hmac[32]`.

**Acceptance criteria:**

1. The HMAC is computed over the raw CBOR bytes using `phone_psk`.
2. `phone_key_hint` (2 bytes, big-endian) is prepended.
3. The 32-byte HMAC is appended.

---

### PT-0405  Gateway public key encryption

**Priority:** Must  
**Source:** ble-pairing-protocol.md §6.3

**Description:**  
The tool MUST encrypt the authenticated request using ECDH + HKDF-SHA256 + AES-256-GCM: generate an ephemeral X25519 keypair, convert `gw_public_key` (Ed25519) to X25519 public key, derive `shared_secret` via X25519, derive `aes_key` via HKDF-SHA256 (salt = `gateway_id`, info = `"sonde-node-pair-v1"`, len = 32), encrypt with AES-256-GCM (AAD = `gateway_id`), and assemble the encrypted payload as `eph_public[32] ‖ nonce[12] ‖ ciphertext`.

**Acceptance criteria:**

1. A fresh ephemeral X25519 keypair is generated for each provisioning attempt.
2. Ed25519 → X25519 conversion rejects low-order points.  *(This criterion overlaps with PT-0902, which is the authoritative requirement for Ed25519 → X25519 low-order point rejection.)*
3. The HKDF salt is `gateway_id` and info is `"sonde-node-pair-v1"`.
4. AES-256-GCM AAD is `gateway_id`.
5. The encrypted payload assembles as `eph_public ‖ nonce ‖ ciphertext`.

---

### PT-0406  Encrypted payload size validation

**Priority:** Must  
**Source:** ble-pairing-protocol.md §11

**Description:**  
The encrypted payload MUST NOT exceed 202 bytes (the ESP-NOW frame budget). If the PairingRequest CBOR is too large, the tool MUST reject the request and report the size constraint to the operator **before** attempting BLE transmission.

**Acceptance criteria:**

1. If the encrypted payload exceeds 202 bytes, the tool does not write to BLE.
2. The error message includes the current size and the 202-byte limit.

---

### PT-0407  NODE_PROVISION transmission and acknowledgement

**Priority:** Must  
**Source:** ble-pairing-protocol.md §6.6, §6.7

**Description:**  
The tool MUST assemble the `NODE_PROVISION` body as `node_key_hint[2] ‖ node_psk[32] ‖ rf_channel[1] ‖ payload_len[2, BE u16] ‖ encrypted_payload`, write it to the Node Command characteristic, and wait for `NODE_ACK` (timeout: 5 s).

**Acceptance criteria:**

1. `NODE_ACK(0x00)` → success, display `node_id`, `node_key_hint`, and `rf_channel`.
2. `NODE_ACK(0x01)` → "node already paired — hold pairing button during boot to factory reset".
3. `NODE_ACK(0x02)` → "node storage error".
4. Timeout (5 s) → "node did not respond".

---

### PT-0408  Node PSK zeroing

**Priority:** Must  
**Source:** security.md §2.1

**Description:**  
After a successful `NODE_PROVISION` write, the tool MUST zero `node_psk` from memory via `zeroize::Zeroizing`. The tool has no further use for it. All ephemeral keys, shared secrets, and derived AES keys from PT-0405 MUST also be zeroed.

**Acceptance criteria:**

1. `node_psk` is zeroed after the `NODE_PROVISION` write succeeds.
2. All intermediate cryptographic material is zeroed after use.
3. No node PSK values are persisted to disk.

---

## 7  Error handling

### PT-0500  Error classification

**Priority:** Must  
**Source:** Product requirements §4.3

**Description:**  
The tool MUST distinguish and report three error categories: **device-level** (BLE adapter not found, Bluetooth disabled, device out of range), **transport-level** (connection dropped, MTU negotiation failed, GATT write failed, indication timeout), and **protocol-level** (signature verification failed, decryption failed, `ERROR` response received, already paired).

**Acceptance criteria:**

1. Each error message identifies the error category.
2. Each error message includes a suggested next step for the operator.

---

### PT-0501  Actionable error messages

**Priority:** Must  
**Source:** Product requirements §4.3

**Description:**  
Every error message MUST include a suggested operator action. Examples: "Bluetooth is disabled — enable Bluetooth in system settings", "Registration window not open — ask the operator to hold the gateway button for 2 seconds", "Node already paired — hold the pairing button during boot to factory reset before re-pairing".

**Acceptance criteria:**

1. No error message consists solely of a code or internal identifier.
2. Every error message includes at least one actionable sentence.

---

### PT-0502  No partial state on failure

**Priority:** Must  
**Source:** Product requirements §4.3

**Description:**  
If any step of the pairing flow fails after a BLE connection is established, the tool MUST disconnect cleanly from the BLE peripheral, MUST NOT persist any partial pairing state, and MUST leave the local pairing store unchanged from its pre-attempt state.

**Acceptance criteria:**

1. A failed Phase 1 does not persist a `phone_psk` or `gw_public_key`.
2. A failed Phase 2 does not persist any node-related data.
3. The BLE connection is released on every error path.

---

## 8  Idempotency and safety

### PT-0600  Re-run safety

**Priority:** Must  
**Source:** Product requirements §4.4

**Description:**  
Re-running the tool MUST NOT corrupt existing paired devices. Re-pairing with the same gateway MUST either succeed (if the registration window is open) or fail with `ERROR(0x03)` — in either case, existing local state MUST remain valid. Re-provisioning a node that is already paired (without holding the pairing button) MUST result in `NODE_ACK(0x01)` and no state change on either side.

**Acceptance criteria:**

1. Repeated Phase 1 against the same gateway does not corrupt local state.
2. Repeated Phase 2 against an already-paired node produces `NODE_ACK(0x01)`.

---

### PT-0601  Already-paired detection

**Priority:** Should  
**Source:** Product requirements §4.4

**Description:**  
The tool SHOULD check local state before initiating Phase 1 and warn the operator if a gateway identity is already persisted. The operator MUST be able to proceed anyway (re-pair) or cancel.

**Acceptance criteria:**

1. If a `gw_public_key` is already stored, the tool warns before starting Phase 1.
2. The operator can choose to proceed or cancel.

---

## 9  User experience

### PT-0700  Minimum UI surface

**Priority:** Must  
**Source:** Product requirements §5.2

**Description:**  
The UI MUST provide at least: scan toggle (start/stop), device list (with service type and name), device select, pair action, node ID input (Phase 2 only), status area (current phase and progress), and error display (actionable messages).

**Acceptance criteria:**

1. All listed UI elements are present and functional.
2. The UI does not include features beyond what is listed here (no management, monitoring, or telemetry).

---

### PT-0701  Phase indication

**Priority:** Must  
**Source:** Product requirements §5.3

**Description:**  
The UI MUST clearly indicate the current phase at all times: idle ("Ready to scan"), scanning, connecting, authenticating (Phase 1), registering (Phase 1), provisioning (Phase 2), complete, or error.

**Acceptance criteria:**

1. The current phase is displayed at all times.
2. Phase transitions are immediate and unambiguous.

---

### PT-0702  Verbose diagnostic mode

**Priority:** Must  
**Source:** Product requirements §5.3

**Description:**  
The tool MUST support an opt-in verbose mode that displays raw BLE events, message types sent/received (by name), cryptographic operation outcomes (never key material), and timing information. This mode MUST NOT be the default.

**Acceptance criteria:**

1. Verbose mode is disabled by default.
2. Verbose mode is activated by an explicit toggle or flag.
3. Key material (PSKs, private keys, shared secrets) never appears in verbose output.

---

## 10  Persistence and local state

### PT-0800  Pairing store contents

**Priority:** Must  
**Source:** ble-pairing-protocol.md §5

**Description:**  
The tool MUST persist the following artifacts after a successful Phase 1: `gw_public_key` (32 bytes), `gateway_id` (16 bytes), `phone_psk` (32 bytes), `phone_key_hint` (2 bytes), `rf_channel` (1 byte), and `phone_label` (operator-supplied string).

**Acceptance criteria:**

1. All listed fields are persisted after Phase 1 completes.
2. The stored values round-trip correctly (read back matches written values).

---

### PT-0801  Platform-appropriate secure storage

**Priority:** Should  
**Source:** security.md §2.1

**Description:**  
PSK material SHOULD be stored using platform-appropriate secure storage: Android Keystore or encrypted SharedPreferences on Android; DPAPI-protected storage or user-profile directory with restricted permissions on Windows.

**Acceptance criteria:**

1. On Android, `phone_psk` is protected by the Android Keystore or equivalent.
2. On Windows, `phone_psk` is stored in `%APPDATA%\sonde\` with restricted file permissions.

---

### PT-0802  Storage abstraction

**Priority:** Must  
**Source:** Product requirements §3.2

**Description:**  
Storage MUST be behind a `PairingStore` trait so that platform-specific secure storage can be injected and tests can use an in-memory implementation.

**Acceptance criteria:**

1. A `PairingStore` trait (or equivalent) exists with read/write/clear operations.
2. Tests use an in-memory implementation.
3. Platform implementations exist for Windows and Android.

---

### PT-0803  Corruption handling

**Priority:** Must  
**Source:** Product requirements §6.1

**Description:**  
If the pairing store is corrupted or cannot be read, the tool MUST report the error clearly, MUST offer to reset the store (delete and re-create), and MUST NOT crash or enter an undefined state.

**Acceptance criteria:**

1. A corrupted store produces a clear error message.
2. The operator can choose to reset the store.
3. The tool does not panic or abort.

---

### PT-0804  No node PSK persistence

**Priority:** Must  
**Source:** security.md §2.1

**Description:**  
The tool MUST NOT persist `node_psk` values after successful provisioning. The `node_psk` is generated, transmitted to the node, embedded in the encrypted payload, and then zeroed from tool memory.

**Acceptance criteria:**

1. No `node_psk` appears in the pairing store after provisioning.
2. No `node_psk` appears in log files.

---

## 11  Security

### PT-0900  No key material in logs

**Priority:** Must  
**Source:** security.md §2.1

**Description:**  
`phone_psk`, `node_psk`, ephemeral private keys, shared secrets, and derived AES keys MUST NOT appear in default log output. Verbose mode MAY log key lengths and operation outcomes but MUST NOT log key bytes.

**Acceptance criteria:**

1. Default log output contains no key material.
2. Verbose log output contains no key material.

---

### PT-0901  CSPRNG for all randomness

**Priority:** Must  
**Source:** security.md §2.1

**Description:**  
All random values (challenges, ephemeral keys, node PSKs, AES-GCM nonces) MUST be generated via `getrandom::fill()` (OS CSPRNG).

**Acceptance criteria:**

1. No use of `rand::rng()` for cryptographic randomness.
2. All random generation calls trace to `getrandom`.

---

### PT-0902  Ed25519 ↔ X25519 conversion safety

**Priority:** Must  
**Source:** ble-pairing-protocol.md §5.5

**Description:**  
When converting the gateway's Ed25519 public key to X25519 for ECDH, the tool MUST reject the conversion if the resulting X25519 public key is a low-order point (all-zero or small-order Curve25519 points).

**Acceptance criteria:**

1. Conversion of an Ed25519 key that maps to a low-order X25519 point returns an error.
2. The error is reported to the operator as "invalid gateway public key".

---

### PT-0903  Clearly non-zero test keys

**Priority:** Must  
**Source:** Project conventions (`.github/copilot-instructions.md`)

**Description:**  
Test code MUST use clearly non-zero keys (e.g., `[0x42u8; 32]`), not `[0u8; 32]`, to avoid normalizing insecure patterns.

**Acceptance criteria:**

1. No test uses `[0u8; 32]` as a PSK or key value.

---

### PT-0904  BLE pairing mode enforcement

**Priority:** Must  
**Source:** security.md §2; ble-pairing-tool-design.md §9.1, §9.2; PT-0106

**Description:**  
The BLE link between the phone and modem MUST be established using LE Secure Connections (LESC) with authenticated pairing (Numeric Comparison). Unauthenticated pairing modes (Just Works, passkey-only without confirmation) MUST NOT be accepted because the pairing exchange carries PSK material that an active MITM could intercept. The transport abstraction MUST expose an explicit, observable signal indicating which pairing method was actually negotiated with the OS BLE stack (e.g., an enum such as `NumericComparison`, `JustWorks`). This signal MUST be available both to the application logic and to the test harness.

On desktop platforms where the BLE library delegates pairing entirely to the OS stack (e.g. btleplug on WinRT, BlueZ, CoreBluetooth), the transport MAY report `None` for the pairing method to indicate that LESC enforcement is OS-managed and not observable by the application. The `enforce_lesc` check MUST treat `None` as acceptable (OS-enforced security), `NumericComparison` as acceptable, and all other values (including `Unknown` and `JustWorks`) as failures that trigger an immediate disconnect.

**Acceptance criteria:**

1. **Precondition:** the modem is configured for LESC with `DisplayYesNo` I/O capability (verified by modem validation, not the pairing tool).
2. The BLE transport interface exposes a pairing-complete hook that includes the resolved pairing method, with at least `NumericComparison` and `JustWorks` as distinguishable values, plus `None` for OS-managed pairing.
3. The phone verifies that the negotiated pairing method is Numeric Comparison **or** `None` (OS-enforced) before proceeding.
4. A Just Works fallback or `Unknown` pairing method is treated as a connection failure, not a silent degradation.
5. The mock BLE transport can be configured to report different pairing methods, and tests for PT-0904 cover success (`NumericComparison`, `None`) and failure (`JustWorks`, `Unknown`) cases.

## 12  Non-functional requirements

### PT-1000  Transient BLE failure tolerance

**Priority:** Must

**Description:**  
The tool MUST tolerate transient BLE failures (momentary signal loss, adapter reset) without requiring application restart. After a connection failure, the tool MUST return to the scanning/idle state and allow the operator to retry.

**Acceptance criteria:**

1. After a connection failure, the tool returns to idle state.
2. The operator can start a new scan without restarting the application.

---

### PT-1001  No resource leaks on failure

**Priority:** Must  
**Source:** Product requirements §7.1

**Description:**  
BLE connections, GATT subscriptions, and platform BLE resources MUST be released on every error path, not only on the success path.

**Acceptance criteria:**

1. No BLE connections remain open after a failed pairing attempt.
2. No GATT subscriptions remain active after disconnect.

---

### PT-1002  Deterministic timeouts

**Priority:** Must  
**Source:** ble-pairing-protocol.md §3.4, §5, §6

**Description:**  
All timeouts MUST be deterministic and explicitly configured to the following values: `GW_INFO_RESPONSE` 45 s, `PHONE_REGISTERED` 30 s, `NODE_ACK` 5 s, BLE scan default 30 s, BLE connection establishment 10 s.

**Acceptance criteria:**

1. Each timeout matches the specified value.
2. Timeout expiry produces a clear error message identifying which operation timed out.

---

### PT-1003  No implicit retries

**Priority:** Must  
**Source:** Product requirements §7.2

**Description:**  
The tool MUST NOT silently retry failed protocol operations. If a write or indication times out, the tool reports the failure and lets the operator decide whether to retry. BLE-level connection retries by the platform stack are acceptable.

**Acceptance criteria:**

1. No protocol-level operation is retried without operator action.
2. Timeout and failure messages are shown to the operator.

---

### PT-1004  Reusable core

**Priority:** Must  
**Source:** Product requirements §7.3

**Description:**  
The pairing state machine and cryptographic logic MUST be usable from the Tauri-based UI, a headless CLI (future), and integration tests with mocked BLE transport.

**Acceptance criteria:**

1. The core crate has no UI or platform dependencies.
2. The core crate is used by at least two consumers: the application frontend and the test suite.

## 13  Cryptographic requirements

### PT-1100  Required primitives

**Priority:** Must  
**Source:** ble-pairing-protocol.md §5, §6

**Description:**  
The tool MUST implement the following cryptographic primitives:

| Primitive | Usage |
|---|---|
| Ed25519 signature verification | Verify gateway identity (Phase 1) |
| X25519 ECDH | Key agreement (Phase 1 and Phase 2) |
| Ed25519 → X25519 conversion | Convert gateway public key for ECDH |
| HKDF-SHA256 | Derive AES key from ECDH shared secret |
| AES-256-GCM | Decrypt `PHONE_REGISTERED` (Phase 1); encrypt pairing payload (Phase 2) |
| HMAC-SHA256 | Authenticate pairing requests with phone PSK |
| SHA-256 | Derive `key_hint` from PSK |
| CSPRNG | Challenges, ephemeral keys, PSKs, nonces (via `getrandom`) |

**Acceptance criteria:**

1. All listed primitives are implemented and used in the correct protocol contexts.
2. Recommended crates: `ed25519-dalek`, `x25519-dalek`, `hkdf`, `sha2`, `aes-gcm`, `hmac`, `getrandom`.

---

### PT-1101  HKDF parameters

**Priority:** Must  
**Source:** ble-pairing-protocol.md §5.5, §6.4

**Description:**  
HKDF-SHA256 MUST use `gateway_id` (16 bytes) as salt. The info string MUST be `"sonde-phone-reg-v1"` for Phase 1 (phone registration) and `"sonde-node-pair-v1"` for Phase 2 (node pairing payload). Output length MUST be 32 bytes.

**Acceptance criteria:**

1. Phase 1 decryption uses info `"sonde-phone-reg-v1"`.
2. Phase 2 encryption uses info `"sonde-node-pair-v1"`.
3. Both use `gateway_id` as salt and produce 32-byte keys.

---

### PT-1102  AES-GCM AAD

**Priority:** Must  
**Source:** ble-pairing-protocol.md §5.5, §6.3

**Description:**  
Both Phase 1 decryption and Phase 2 encryption MUST use `gateway_id` as the AES-256-GCM Additional Authenticated Data (AAD).

**Acceptance criteria:**

1. AES-256-GCM operations use `gateway_id` as AAD.
2. Decryption with incorrect AAD fails (GCM tag mismatch).

---

### PT-1103  Deterministic CBOR encoding

**Priority:** Must  
**Source:** ble-pairing-protocol.md §6.1

**Description:**  
The PairingRequest CBOR (Phase 2) MUST be encoded per RFC 8949 §4.2: integer keys in ascending order, minimal-length integer encoding, definite-length containers.

**Acceptance criteria:**

1. Encoding the same PairingRequest twice produces identical bytes.
2. Keys appear in order 1, 2, 3, 4, 5, 6.

### PT-1213  Build-type–aware log levels

**Priority:** Must  
**Source:** issue #496

**Description:**  
The pairing tool MUST apply build-type–aware log-level policies: compile-time gating to strip DEBUG/TRACE call-sites in release builds, and a runtime default that differs between debug and release. The `sonde-pair` library crate and the `sonde-pair-ui` application crate both apply compile-time gating via `tracing` Cargo features.

**Acceptance criteria:**

1. In debug builds, the compile-time maximum tracing level is TRACE.
2. In release builds, the compile-time maximum tracing level is INFO (`trace!` and `debug!` call-sites are no-ops).
3. The default `EnvFilter` in the Tauri entry point is `sonde_pair=info,sonde_pair_ui=info` in debug builds and `sonde_pair=warn,sonde_pair_ui=warn` in release builds.
4. `RUST_LOG` overrides the default in both build types (within compile-time limits).
5. The `tracing` crate dependency in both `sonde-pair` and `sonde-pair-ui` specifies `features = ["max_level_trace", "release_max_level_info"]`.
