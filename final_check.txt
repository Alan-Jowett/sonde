   - IKM: shared_secret (32 bytes from X25519)

   ✓ All parameters specified

5. Nonce reuse check:
   - Phone registration: Fresh ephemeral keypair + random nonce per registration
   - Node pairing: Fresh ephemeral keypair + random nonce per node
   ✓ No nonce reuse risk (ephemeral keys provide primary defense)

6. Replay protection:
   - PEER_REQUEST: Timestamp check (±24h) + node_id uniqueness (§7.3 steps 9-10)
   - PEER_ACK: seq field echoes PEER_REQUEST nonce (§7.2)
   ✓ Replay protection specified

7. Authentication layers:
   - Phone→Gateway: Ephemeral ECDH + HMAC with phone PSK
   - Node→Gateway: Ephemeral ECDH + HMAC with phone PSK + frame HMAC with node PSK
   ✓ Multi-layer authentication

8. Key binding:
   - registration_proof binds node PSK to encrypted payload (§7.2)
   - Gateway signature binds gateway public key to challenge (§5.3)
   ✓ Key binding mechanisms present

POTENTIAL ISSUES TO FLAG:

A. registration_proof uses simple hash, not HMAC
   - registration_proof = SHA256(node_psk ‖ encrypted_payload)
   - This is a MAC without HMAC construction
   - However, the encrypted_payload is already integrity-protected by AES-GCM
   - The registration_proof serves as a commitment, not primary integrity check
   - VERDICT: Acceptable design, but worth noting

B. No explicit endianness for HMAC/hash inputs
   - Document specifies "all multi-byte integers are big-endian" (§1.3)
   - But HMAC and hash inputs are byte strings, not integers
   - HMAC(key, data) where data is "header ‖ cbor_payload" - byte concatenation
   - VERDICT: Spec is clear enough (byte concatenation is unambiguous)

C. Gateway Ed25519 keypair generation not fully specified
   - §2.1 says "Generated on first gateway startup"
   - No specification of seed generation or key generation algorithm
   - VERDICT: Should reference standard Ed25519 key generation (RFC 8032)
