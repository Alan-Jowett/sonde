# Gateway Requirements Specification

> **Document status:** Draft  
> **Source:** Derived from [README.md](../README.md) (design-phase specification).  
> **Scope:** This document covers the Sonde **gateway** component only. Node firmware requirements are out of scope.

---

## 1  Definitions

| Term | Definition |
|---|---|
| **Node** | A remote sensor device running Sonde firmware and executing BPF programs. |
| **Gateway** | The central component that manages nodes, distributes programs, and processes application data. |
| **Resident program** | A BPF program stored in flash on a node for long-term, scheduled execution. |
| **Ephemeral program** | A one-shot BPF program sent to a node for immediate execution and discarded afterward. |
| **Program hash** | A cryptographic hash that uniquely identifies the content of a BPF program. |
| **CBOR** | Concise Binary Object Representation — the encoding format for all control-plane messages. |
| **ESP-NOW** | A connectionless 802.11 protocol used as the radio transport in the reference implementation. |
| **Map** | A BPF key-value data structure persisted in node sleep-persistent memory. |
| **Nonce** | A 64-bit random value generated by a node each wake cycle for replay protection. |

---

## 2  Requirement format

Each requirement uses the following fields:

- **ID** — Unique identifier (`GW-XXXX`).
- **Title** — Short name.
- **Description** — What the gateway must do.
- **Acceptance criteria** — Observable, testable conditions that confirm the requirement is met.
- **Priority** — MoSCoW: **Must**, **Should**, **May**.
- **Source** — Section of the README that motivates the requirement.

---

## 3  Protocol and communication

### GW-0100  Node-initiated communication

**Priority:** Must  
**Source:** README § Node-gateway protocol

**Description:**  
The gateway MUST NOT initiate communication with nodes. All communication is node-initiated; the gateway only responds to incoming messages.

**Acceptance criteria:**

1. The gateway never transmits a frame unless it has received a message from the target node in the current wake cycle.
2. If no nodes contact the gateway, no outbound traffic is generated.

---

### GW-0101  CBOR message encoding

**Priority:** Must  
**Source:** README § Node-gateway protocol

**Description:**  
All control-plane messages produced by the gateway MUST be encoded in CBOR.

**Acceptance criteria:**

1. Every outbound control-plane message is valid CBOR as defined by RFC 8949.
2. The gateway correctly decodes all valid inbound CBOR messages from nodes.
3. The gateway rejects malformed CBOR with an appropriate error and does not crash.

---

### GW-0102  Wake handshake — WAKE reception

**Priority:** Must  
**Source:** README § Wake handshake

**Description:**  
The gateway MUST accept `WAKE` messages containing the fields: `node_id`, `nonce`, `firmware_abi_version`, `program_hash`, and `battery_mv`.

**Acceptance criteria:**

1. The gateway parses all five fields from a valid `WAKE` message.
2. The gateway rejects `WAKE` messages missing any required field.
3. Parsed values are made available for command-selection logic.

---

### GW-0103  Wake handshake — COMMAND response

**Priority:** Must  
**Source:** README § Wake handshake

**Description:**  
The gateway MUST respond to every valid, authenticated `WAKE` message with exactly one `COMMAND` message. The response MUST include the nonce from the originating `WAKE` message.

**Acceptance criteria:**

1. For every valid `WAKE`, the gateway emits exactly one `COMMAND` response.
2. The `nonce` in the response matches the `nonce` from the `WAKE`.
3. The `command_type` field is one of the defined command types (see GW-0200–GW-0205).

---

### GW-0104  Frame size constraint (reference implementation)

**Priority:** Must  
**Source:** README § Reference implementation

**Description:**  
When using ESP-NOW transport, all gateway-originated frames MUST fit within 250 bytes total, yielding approximately 210 bytes of payload after authentication overhead (32-byte HMAC + 8-byte nonce).

**Acceptance criteria:**

1. No outbound frame exceeds 250 bytes.
2. Payload content does not exceed the usable payload budget after authentication fields are appended.

---

## 4  Command set

### GW-0200  NOP command

**Priority:** Must  
**Source:** README § Commands

**Description:**  
The gateway MUST support the `NOP` command, which instructs the node to proceed directly to BPF execution without any state change.

**Acceptance criteria:**

1. When the gateway determines no update or action is needed, it responds with `NOP`.
2. The `NOP` command carries no additional payload beyond the nonce.

---

### GW-0201  UPDATE_PROGRAM command

**Priority:** Must  
**Source:** README § Commands, § Chunked program transfer

**Description:**  
The gateway MUST support the `UPDATE_PROGRAM` command to signal that a new resident BPF program is available. This initiates a chunked transfer sequence (see GW-0300).

**Acceptance criteria:**

1. The gateway issues `UPDATE_PROGRAM` when the node's reported `program_hash` differs from the gateway's intended program for that node.
2. The command includes metadata sufficient for the node to begin the chunked transfer.

---

### GW-0202  RUN_EPHEMERAL command

**Priority:** Must  
**Source:** README § Commands, § Two classes of BPF programs

**Description:**  
The gateway MUST support the `RUN_EPHEMERAL` command, delivering a one-shot BPF program for immediate execution on the node.

**Acceptance criteria:**

1. The ephemeral program bytecode is included in (or immediately follows) the command.
2. The delivered program has been verified against the ephemeral verifier profile (see GW-0401) before transmission.
3. If the program exceeds the ephemeral size budget (2 KB recommended), the gateway rejects the dispatch.

---

### GW-0203  UPDATE_SCHEDULE command

**Priority:** Must  
**Source:** README § Commands, § Schedule model

**Description:**  
The gateway MUST support the `UPDATE_SCHEDULE` command to set a new base wake interval on a node.

**Acceptance criteria:**

1. The command includes the new base interval in seconds.
2. The gateway records the updated schedule for the target node.

---

### GW-0204  REBOOT command

**Priority:** Must  
**Source:** README § Commands

**Description:**  
The gateway MUST support the `REBOOT` command, instructing the node to restart its firmware.

**Acceptance criteria:**

1. The gateway can issue `REBOOT` to any known node.
2. The command carries no payload beyond the nonce and command type.

---

### GW-0205  APP_MSG command

**Priority:** Must  
**Source:** README § Commands

**Description:**  
The gateway MUST support the `APP_MSG` command, delivering an opaque blob that is passed into the BPF program's execution context on the node.

**Acceptance criteria:**

1. The gateway can attach an arbitrary byte blob to the `APP_MSG` command.
2. The blob content is opaque to the gateway; no interpretation or validation of its content is performed.

---

## 5  Chunked program transfer

### GW-0300  Chunk serving

**Priority:** Must  
**Source:** README § Chunked program transfer

**Description:**  
The gateway MUST serve program chunks in response to `GET_CHUNK` requests from nodes. Transfer is node-driven and uses stop-and-wait.

**Acceptance criteria:**

1. The gateway responds to `GET_CHUNK { nonce, chunk_index }` with `CHUNK { nonce, chunk_index, chunk_data }`.
2. The nonce in the chunk response matches the nonce from the request.
3. Each chunk fits within the frame size constraint (GW-0104).
4. All chunks for a given program, when reassembled, produce a byte-identical copy of the original program.

---

### GW-0301  Transfer resumption

**Priority:** Must  
**Source:** README § Chunked program transfer

**Description:**  
The gateway MUST support transfer resumption. If a node restarts the transfer from `chunk_index = 0` (e.g., after power loss), the gateway MUST serve the program from the beginning without error.

**Acceptance criteria:**

1. Repeated requests for `chunk_index = 0` from the same node do not cause errors or state corruption.
2. A node can complete the transfer across multiple wake cycles.

---

### GW-0302  Program acknowledgement

**Priority:** Must  
**Source:** README § Chunked program transfer

**Description:**  
The gateway MUST accept `PROGRAM_ACK { nonce, version }` messages from nodes indicating successful program installation.

**Acceptance criteria:**

1. Upon receiving `PROGRAM_ACK`, the gateway updates its record of the node's current program.
2. Subsequent `WAKE` messages from the node with the new `program_hash` no longer trigger `UPDATE_PROGRAM`.

---

## 6  BPF program management

### GW-0400  Program compilation

**Priority:** Must  
**Source:** README § Architecture ("compile, verify")

**Description:**  
The gateway MUST be capable of compiling BPF source into BPF bytecode (ELF format) suitable for node execution.

**Acceptance criteria:**

1. The gateway accepts BPF program source and produces a valid BPF ELF binary.
2. The output conforms to the helper API and ABI expected by node firmware.

---

### GW-0401  Program verification (Prevail)

**Priority:** Must  
**Source:** README § Verification (Prevail)

**Description:**  
The gateway MUST verify all BPF programs using the Prevail verifier before distributing them to nodes. Two verification profiles exist:

| Property | Resident | Ephemeral |
|---|---|---|
| Loops | Bounded | None or tightly bounded |
| Map access | Read/write | Read-only |
| Instruction budget | Larger | Small |
| Helper set | Full | Limited |
| Side effects | Allowed | None |

**Acceptance criteria:**

1. No program is distributed to a node unless it passes verification.
2. Resident programs are verified against the resident profile.
3. Ephemeral programs are verified against the ephemeral profile.
4. Programs that fail verification are rejected with a clear diagnostic indicating the reason.

---

### GW-0402  Program identity by content hash

**Priority:** Must  
**Source:** README § Wake handshake

**Description:**  
The gateway MUST identify programs by their content hash, not by version numbers. The program's identity is its hash.

**Acceptance criteria:**

1. The gateway computes a hash over the compiled program bytecode.
2. The gateway compares this hash with the `program_hash` reported by a node to determine whether an update is needed.
3. Identical program content always produces the same hash.

---

### GW-0403  Program size enforcement

**Priority:** Should  
**Source:** README § Reference implementation

**Description:**  
The gateway SHOULD enforce maximum program sizes: 4 KB for resident programs and 2 KB for ephemeral programs (recommended limits for the reference implementation).

**Acceptance criteria:**

1. Programs exceeding the configured size limit are rejected at compilation or distribution time.
2. The size limits are configurable to accommodate different hardware targets.

---

## 7  Application data

### GW-0500  APP_DATA reception

**Priority:** Must  
**Source:** README § Application data

**Description:**  
The gateway MUST receive and store `APP_DATA { nonce, blob }` messages from nodes. The blob content is opaque to the gateway's transport layer.

**Acceptance criteria:**

1. The gateway accepts `APP_DATA` messages from authenticated nodes.
2. The blob is stored or forwarded without modification.
3. The gateway associates each blob with the originating `node_id` and a reception timestamp.

---

### GW-0501  Application data forwarding

**Priority:** Should  
**Source:** README § Application data

**Description:**  
The gateway SHOULD provide a mechanism (API, callback, or event system) for application-level code to consume received `APP_DATA` blobs.

**Acceptance criteria:**

1. An application running on the gateway can register to receive `APP_DATA` for one or more nodes.
2. Blobs are delivered to the application in the order received from each node.

---

## 8  Authentication and security

### GW-0600  HMAC-SHA256 message authentication

**Priority:** Must  
**Source:** README § Authentication

**Description:**  
The gateway MUST authenticate all inbound messages using HMAC-SHA256 and MUST append a valid HMAC-SHA256 tag to all outbound messages. The HMAC covers both header and payload.

**Acceptance criteria:**

1. Inbound messages with invalid HMAC are silently discarded.
2. Every outbound message includes a 32-byte HMAC-SHA256 tag computed over header + payload using the target node's key.
3. The gateway never processes an unauthenticated message beyond HMAC validation.

---

### GW-0601  Per-node key management

**Priority:** Must  
**Source:** README § Key provisioning

**Description:**  
The gateway MUST maintain a mapping of `node_id` to 256-bit pre-shared key. Each node has a unique key. There is no runtime key exchange.

**Acceptance criteria:**

1. The gateway can look up the correct key for any registered node by `node_id`.
2. Messages from unknown `node_id` values (no key on file) are discarded.
3. The key store supports addition and removal of nodes.

---

### GW-0602  Replay protection — nonce sliding window

**Priority:** Must  
**Source:** README § Replay protection

**Description:**  
The gateway MUST implement per-node replay protection using a sliding window of seen nonces. Messages with previously seen nonces MUST be rejected.

**Acceptance criteria:**

1. A message replayed with the same nonce as a previously accepted message is rejected.
2. The sliding window is sized to tolerate reasonable network reordering.
3. Nonce state is maintained per node.

---

### GW-0603  Authentication overhead budget

**Priority:** Must  
**Source:** README § Overhead

**Description:**  
Authentication adds 40 bytes per frame (32-byte HMAC + 8-byte nonce). The gateway MUST account for this overhead when constructing frames.

**Acceptance criteria:**

1. The gateway reserves 40 bytes of every outbound frame for authentication fields.
2. Payload sizing logic correctly subtracts authentication overhead from the maximum frame size.

---

## 9  Node management

### GW-0700  Node registry

**Priority:** Must  
**Source:** README § Key provisioning, § Gateway failover

**Description:**  
The gateway MUST maintain a registry of known nodes, including at minimum: `node_id`, pre-shared key, assigned program (hash), and current schedule.

**Acceptance criteria:**

1. The gateway can enumerate all registered nodes.
2. Each node entry contains the fields listed above.
3. The registry is persistent across gateway restarts.

---

### GW-0701  Stale program detection

**Priority:** Must  
**Source:** README § Wake handshake

**Description:**  
On receiving a `WAKE` message, the gateway MUST compare the node's reported `program_hash` with the intended program and issue `UPDATE_PROGRAM` if they differ.

**Acceptance criteria:**

1. If `program_hash` matches the intended program, the gateway does not issue `UPDATE_PROGRAM`.
2. If `program_hash` differs, the gateway issues `UPDATE_PROGRAM` to begin a chunked transfer.

---

### GW-0702  Battery level tracking

**Priority:** Should  
**Source:** README § Wake handshake

**Description:**  
The gateway SHOULD record the `battery_mv` value from each `WAKE` message for monitoring and operational visibility.

**Acceptance criteria:**

1. The most recent `battery_mv` reading is stored per node.
2. Historical battery data is available for trend analysis.

---

### GW-0703  Firmware ABI version awareness

**Priority:** Must  
**Source:** README § Wake handshake, § Helper API

**Description:**  
The gateway MUST inspect the `firmware_abi_version` from `WAKE` messages and ensure that distributed programs are compatible with the node's ABI.

**Acceptance criteria:**

1. The gateway records each node's `firmware_abi_version`.
2. The gateway does not distribute a program compiled for an incompatible ABI version.
3. An ABI mismatch is reported as a clear error or warning.

---

## 10  Operational requirements

### GW-1000  Gateway failover / replaceability

**Priority:** Must  
**Source:** README § Gateway failover

**Description:**  
The gateway MUST be replaceable by another instance provisioned with the same key database. The gateway's identity is defined solely by its knowledge of node keys and program assignments — not by any hardware or network identity.

**Acceptance criteria:**

1. A second gateway instance loaded with the same key database and program set can serve nodes without node-side reconfiguration.
2. Nodes continue normal operation after a gateway swap.

---

### GW-1001  Exportable / importable state

**Priority:** Should  
**Source:** README § Gateway failover

**Description:**  
The gateway SHOULD support exporting and importing its full state (node registry, keys, program assignments, schedules) to enable failover and backup.

**Acceptance criteria:**

1. The gateway can export its state to a portable format.
2. A fresh gateway instance can import the state and resume operation.
3. The export format does not leak keys in plaintext unless explicitly requested.

---

### GW-1002  Graceful handling of unknown nodes

**Priority:** Must  
**Source:** README § Authentication

**Description:**  
The gateway MUST silently discard messages from nodes whose `node_id` is not in the key registry. No error response is sent (to avoid information leakage).

**Acceptance criteria:**

1. Messages from unregistered nodes produce no outbound traffic.
2. The event is logged for operational monitoring.
3. No internal state is altered by messages from unknown nodes.

---

### GW-1003  Concurrent node handling

**Priority:** Should  
**Source:** Implied by distributed sensor network operation.

**Description:**  
The gateway SHOULD handle multiple simultaneous node wake events without serializing all processing.

**Acceptance criteria:**

1. Two or more nodes sending `WAKE` messages concurrently both receive timely `COMMAND` responses.
2. Per-node state is isolated; processing one node's request does not corrupt another's.
