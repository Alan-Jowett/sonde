# BLE Pairing Protocol Specification

> **Document status:** Draft вҖ” under review
> **Scope:** Wire-level protocol for BLE-mediated pairing between a mobile phone app, sonde nodes, and the sonde gateway.
> **Audience:** Implementers building the mobile pairing app, the node firmware BLE handler, or the gateway pairing subsystem.
> **Related:** [pairing-protocol.md](pairing-protocol.md) (legacy USB), [protocol.md](protocol.md), [security.md](security.md)

---

## 1  Overview

A mobile phone acts as a pairing agent.  It communicates with the gateway and node over BLE to provision key material and register nodes.  The protocol has two phases:

1. **PhoneвҖ“gateway pairing** (one-time) вҖ” the phone obtains the gateway's public key and receives a phone PSK.
2. **Node pairing** (per node) вҖ” the phone provisions a node PSK over BLE, and the node relays an encrypted registration payload to the gateway over ESP-NOW.

### 1.1  Actors

| Actor | Transport | Role |
|-------|-----------|------|
| Phone | BLE (to gateway and node) | Generates node PSKs, authenticates pairing requests with its phone PSK, encrypts payloads with gateway public key. |
| Node | BLE (from phone), ESP-NOW (to gateway) | Stores node PSK, relays opaque encrypted payload to gateway. |
| Gateway | BLE (to phone), ESP-NOW (from node) | Issues phone PSKs, decrypts and validates pairing requests, registers nodes. |

### 1.2  Cryptographic primitives

| Primitive | Parameters | Usage |
|-----------|------------|-------|
| HMAC-SHA256 | 32-byte key, 32-byte output | Phone authenticates pairing requests.  Node authenticates ESP-NOW frames. |
| Ed25519 | 32-byte public key, 64-byte signature | Gateway keypair (public key distributed for X25519 conversion). |
| X25519 | 32-byte keys | ECDH key agreement (ephemeral вҶ” gateway). |
| HKDF-SHA256 | Variable-length IKM, salt, info | Derives AES key from ECDH shared secret. |
| AES-256-GCM | 32-byte key, 12-byte nonce, 16-byte tag | Encrypts payloads for the gateway. |
| SHA-256 | 32-byte output | key_hint derivation. |

The node only uses HMAC-SHA256 and SHA-256 (already implemented).  Ed25519, X25519, HKDF-SHA256, and AES-256-GCM are new cryptographic dependencies required on the phone and gateway only.

### 1.3  Notation

- All multi-byte integers are **big-endian** unless stated otherwise.
- `вҖ–` denotes byte concatenation.
- `[N]` after a field name means exactly N bytes.
- `HMAC(key, data)` means HMAC-SHA256.
- `SHA256(data)` means SHA-256.

## 2  Key material

### 2.1  Gateway Ed25519 keypair

Generated on first gateway startup.  Private key stored encrypted at rest (GW-0601a master key).

| Item | Size | Description |
|------|------|-------------|
| `gw_private_key` | 32 bytes | Ed25519 seed (the 32-byte private keying material; never leaves the gateway). |
| `gw_public_key` | 32 bytes | Distributed to phones during Phase 1. |
| `gateway_id` | 16 bytes | Random identifier, generated alongside the keypair.  Used as HKDF salt and AES-GCM AAD. |

The Ed25519 public key is converted to X25519 using the standard birational map (RFC 8032 вҶ’ RFC 7748) whenever ECDH is needed.

### 2.2  Phone PSK

Each authorized phone receives a unique 256-bit PSK from the gateway.

| Item | Size | Derivation |
|------|------|------------|
| `phone_psk` | 32 bytes | Generated by gateway from OS CSPRNG. |
| `phone_key_hint` | 2 bytes | `SHA256(phone_psk)[30..32]` вҖ” big-endian u16. |

The gateway stores each phone PSK with: `phone_key_hint`, label (UTF-8, max 64 bytes), issued-at timestamp (u64 seconds), and status (active / revoked).

### 2.3  Node PSK

Generated by the phone for each node.

| Item | Size | Derivation |
|------|------|------------|
| `node_psk` | 32 bytes | Generated by phone from OS CSPRNG. |
| `node_key_hint` | 2 bytes | `SHA256(node_psk)[30..32]` вҖ” big-endian u16. |

The key_hint derivation (`SHA256(PSK)[30..32]`) is a provisioning convention shared with the USB pairing protocol.  The radio protocol treats key_hint as an opaque 16-bit lookup optimization вҖ” the derivation method is not enforced at the protocol layer.

## 3  BLE GATT services

### 3.1  Service UUIDs

Two GATT services are defined вҖ” one on the gateway (or its BLE-connected modem), one on the node.

| Service | UUID | Advertised by |
|---------|------|---------------|
| Gateway Pairing Service | `0000FE60-0000-1000-8000-00805F9B34FB` | Gateway |
| Node Provisioning Service | `0000FE50-0000-1000-8000-00805F9B34FB` | Node (in pairing mode) |

### 3.2  Gateway Pairing Service characteristics

| Characteristic | UUID | Properties | Description |
|----------------|------|------------|-------------|
| Gateway Command | `0000FE61-...` | Write, Indicate | Phone writes commands, gateway sends responses as indications. |

### 3.3  Node Provisioning Service characteristics

| Characteristic | UUID | Properties | Description |
|----------------|------|------------|-------------|
| Node Command | `0000FE51-...` | Write, Indicate | Phone writes provision data, node sends ACK as indication. |

> **Note:** UUIDs are provisional; final values will be assigned before v1.0.

### 3.4  MTU and fragmentation

Both services require an ATT MTU вүҘ 247 bytes (negotiated via MTU exchange).  Messages larger than (MTU вҲ’ 3) bytes use **Write Long** (ATT Prepare Write + Execute Write) for writes, and are reassembled from multiple indications on the receiving side using the length-prefixed envelope (В§4).

If MTU negotiation yields < 247, the phone should disconnect and report an error.

### 4.1  PhoneвҖ“gateway message types (Gateway Command characteristic)

|------|-----------|------|-------------|

| Offset | Size | Field   | Description          |
|--------|------|---------|----------------------|
| 0      | 1    | status  | Error code (u8)      |
| 1      | var  | message | UTF-8 diagnostic text |

**Status codes:**

| Code | Meaning |
|------|---------|
| 0x01 | Generic error |
| 0x02 | Registration window closed |
| 0x03 | Already paired |

| Type | Direction | Name | Description |
|------|-----------|------|-------------|
| 0x01 | Phone вҶ’ Node | `NODE_PROVISION` | Provision PSK + encrypted payload. |

### 5.2  REQUEST_GW_INFO (0x01)

**Body: 32 bytes.**

```
Offset  Size  Field
в”Җв”Җв”Җв”Җв”Җв”Җв”Җ в”Җв”Җв”Җв”Җв”Җ в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
0       32    challenge           Random nonce generated by the phone
```

The phone generates a 32-byte random challenge nonce from OS CSPRNG.  The gateway must sign this nonce to prove it holds the Ed25519 private key corresponding to the public key it advertises.

> **Trust-on-first-use (TOFU):**  On the phone's first connection to a gateway, there is no pre-existing trust anchor вҖ” an attacker could present their own keypair and produce a valid signature.  The phone MUST persist `gw_public_key` and `gateway_id` after first successful verification (pin-on-first-use).  Subsequent connections reject any gateway presenting a different public key.  For environments requiring stronger assurance, the operator SHOULD verify the gateway's public key fingerprint out-of-band (e.g., displayed on the gateway's admin interface and confirmed visually on the phone).

### 5.3  GW_INFO_RESPONSE (0x81)

**Body: 112 bytes.**

```
Offset  Size  Field
в”Җв”Җв”Җв”Җв”Җв”Җв”Җ в”Җв”Җв”Җв”Җв”Җ в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
0       32    gw_public_key       Ed25519 public key
32      16    gateway_id          Random gateway identifier
48      64    signature           Ed25519 signature over (challenge вҖ– gateway_id)
```

The gateway signs the concatenation of the phone's challenge nonce (from `REQUEST_GW_INFO`) and its own `gateway_id`.  Including `gateway_id` in the signed payload binds the signature to this specific gateway, preventing an attacker from replaying a signature obtained from a different gateway.

**Phone verification:**

1. Verify `signature` over `(challenge вҖ– gateway_id)` using `gw_public_key`.
2. If verification fails вҶ’ disconnect, report error ("gateway authentication failed").
3. If verification succeeds вҶ’ the phone has confirmed the gateway holds the private key.  Persist `gw_public_key` and `gateway_id`.

### 5.4  REGISTER_PHONE (0x02)

**Body: 33 + label_len bytes.**

```
Offset  Size        Field
в”Җв”Җв”Җв”Җв”Җв”Җв”Җ в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
0       32          ephemeral_pubkey    Phone's ephemeral X25519 public key
32      1           label_len           Length of label (0вҖ“64)
33      label_len   label               UTF-8 phone label
```

The phone generates a fresh ephemeral X25519 keypair for this request.  The ephemeral public key enables the gateway to encrypt the response.

#### 5.4.1  Registration window

The gateway MUST reject `REGISTER_PHONE` requests unless a **registration window** is currently open.  The registration window is opened by an explicit operator action:

- **Physical button** on the gateway (or modem) held for вүҘ2 seconds, OR
- **Admin API call** (planned `OpenPhoneRegistration` gRPC вҖ” not yet defined in `admin.proto`; will be added in a future PR).

The registration window remains open for a configurable duration (default: 120 seconds) and closes automatically.  While closed, `REGISTER_PHONE` receives an `ERROR` response (type `0xFF`) with status `0x02` ("Registration window closed", per В§4.1.1).

This prevents an unauthorized device from silently registering as a pairing agent by simply connecting to the gateway's BLE service.

### 5.5  PHONE_REGISTERED (0x82)

**Body: 12 + ciphertext_len bytes.**

The gateway encrypts the response so the phone PSK is never sent in plaintext (even within the BLE LESC session):

1. Convert `gw_private_key` (Ed25519 seed) to X25519 private key: the Ed25519 private key is stored as a 32-byte seed (the raw keying material from which the expanded secret key is derived).  To obtain the X25519 private key, compute `SHA-512(seed)` and take the first 32 bytes, then clamp per RFC 7748 В§5 (clear bits 0, 1, 2 of the first byte; clear bit 7 and set bit 6 of the last byte).  This matches the procedure used by libsodium's `crypto_sign_ed25519_sk_to_curve25519`.  Implementations using an "expanded" 64-byte Ed25519 secret key (seed вҖ– public) MUST extract the first 32 bytes (the seed) before hashing.
2. `shared_secret` = X25519(gw_x25519_private, ephemeral_pubkey from REGISTER_PHONE).
3. `aes_key` = HKDF-SHA256(ikm = shared_secret, salt = gateway_id, info = `"sonde-phone-reg-v1"`, len = 32).
4. `nonce` = 12 random bytes from OS CSPRNG.
5. `ciphertext вҖ– tag` = AES-256-GCM-Encrypt(key = aes_key, nonce = nonce, plaintext = inner, aad = gateway_id).

Wire format:

```
Offset  Size             Field
в”Җв”Җв”Җв”Җв”Җв”Җв”Җ в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
0       12               nonce           AES-256-GCM nonce
12      ciphertext_len   ciphertext      Encrypted inner + 16-byte GCM tag
```

The **inner plaintext** (36 bytes):

```
Offset  Size  Field
в”Җв”Җв”Җв”Җв”Җв”Җв”Җ в”Җв”Җв”Җв”Җв”Җ в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
0       1     status          0x00 = accepted, 0x01 = rejected
1       32    phone_psk       256-bit phone PSK
33      2     phone_key_hint  BE u16
35      1     rf_channel      WiFi channel (1вҖ“13)
```

### 5.6  Error handling

| Condition | Behavior |
|-----------|----------|
| MTU < 247 | Phone disconnects, reports error. |
| `GW_INFO_RESPONSE` not received within 5 s | Phone disconnects, reports timeout. |
| Gateway signature verification fails | Phone disconnects, reports error ("gateway authentication failed вҖ” possible impersonation"). |
| `PHONE_REGISTERED` not received within 30 s | Phone disconnects, reports timeout (allows time for operator confirmation). |
| `status` = 0x01 (rejected) | Phone reports rejection ("registration window not open вҖ” ask operator to enable pairing"), disconnects. |
| `ERROR` (0xFF) received in response to `REGISTER_PHONE` | Phone reports rejection, disconnects. |
| AES-GCM decryption failure | Phone reports error (ephemeral key mismatch), disconnects. |

### 5.7  Phone persistence

On successful registration, the phone persists:

| Item | Source |
|------|--------|
| `gw_public_key` | From `GW_INFO_RESPONSE` |
| `gateway_id` | From `GW_INFO_RESPONSE` |
| `phone_psk` | From decrypted `PHONE_REGISTERED` |
| `phone_key_hint` | From decrypted `PHONE_REGISTERED` |
| `rf_channel` | From decrypted `PHONE_REGISTERED` |

## 6  Phase 2 вҖ” Node pairing

### 6.1  Prerequisites

The phone has completed Phase 1 and holds: `gw_public_key`, `gateway_id`, `phone_psk`, `phone_key_hint`, `rf_channel`.

### 6.2  User input

| Field | Required | Constraints |
|-------|----------|-------------|
| `node_id` | Yes | UTF-8, 1вҖ“64 bytes.  Human-readable identifier. |
| `sensors` | No | CBOR array of sensor descriptors (schema in В§6.3.1). |

### 6.3  Pairing request construction

The phone builds a **PairingRequest** as a CBOR map with integer keys:

```cbor
{
  1: node_id,           ; tstr (1вҖ“64 bytes)
  2: node_key_hint,     ; uint (u16, 0вҖ“65535)
  3: node_psk,          ; bstr (32 bytes)
  4: rf_channel,        ; uint (1вҖ“13)
  5: sensors,           ; array (optional, see В§6.3.1)
  6: timestamp          ; uint (Unix seconds)
}
```

CBOR encoding MUST use deterministic encoding (RFC 8949 В§4.2): integer keys in ascending order, minimal-length integer encoding, definite-length containers.

#### 6.3.1  Sensor descriptor schema

Each sensor is a CBOR map:

```cbor
{
  1: sensor_type,       ; uint (enum: 1=i2c, 2=adc, 3=gpio, 4=spi)
  2: sensor_id,         ; uint (0вҖ“255, bus address or pin number)
  3: label              ; tstr (optional, max 64 bytes UTF-8, human-readable name)
}
```

Example: `[{1: 1, 2: 0x48, 3: "soil_moisture"}, {1: 2, 2: 3, 3: "battery"}]`

### 6.4  Authentication and encryption

Step 1 вҖ” **Authenticate** with phone PSK:

```
hmac_input    = PairingRequest_cbor_bytes
phone_hmac    = HMAC-SHA256(phone_psk, hmac_input)

authenticated_request =
    phone_key_hint[2]  вҖ–  PairingRequest_cbor_bytes  вҖ–  phone_hmac[32]
```

Total: 2 + len(cbor) + 32 bytes.

Step 2 вҖ” **Encrypt** with gateway public key:

1. Generate ephemeral X25519 keypair: `(eph_private, eph_public)`.
2. Convert `gw_public_key` (Ed25519) to X25519 public key.
3. `shared_secret` = X25519(eph_private, gw_x25519_public).
4. `aes_key` = HKDF-SHA256(ikm = shared_secret, salt = gateway_id, info = `"sonde-node-pair-v1"`, len = 32).
5. `nonce` = 12 random bytes from OS CSPRNG.
6. `ciphertext вҖ– tag` = AES-256-GCM-Encrypt(key = aes_key, nonce = nonce, plaintext = authenticated_request, aad = gateway_id).

**Encrypted pairing payload:**

```
Offset  Size              Field
в”Җв”Җв”Җв”Җв”Җв”Җв”Җ в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
0       32                eph_public          Ephemeral X25519 public key
32      12                nonce               AES-256-GCM nonce
44      variable          ciphertext          Encrypted authenticated_request + 16B GCM tag
```

Total: 44 + len(authenticated_request) + 16 bytes.

### 6.6  NODE_PROVISION (0x01, Phone вҶ’ Node)

**Body: 37 + payload_len bytes.**

```
Offset      Size          Field
в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
```

### 6.7  NODE_ACK (0x81, Node вҶ’ Phone)

**Body: 1 byte.**

## 7  ESP-NOW transport (Node вҶ” Gateway)

### 7.1  PEER_REQUEST (msg_type 0x05, Node вҶ’ Gateway)

Uses the standard ESP-NOW frame format from [protocol.md](protocol.md):

```
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ Header (11 bytes)                       в”Ӯ CBOR payload  в”Ӯ HMAC   в”Ӯ
в”Ӯ key_hint[2] вҖ– msg_type[1] вҖ– nonce[8]   в”Ӯ               в”Ӯ [32]   в”Ӯ
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

| Header field | Value |
|-------------|-------|
| `key_hint` | Node's key_hint (2 bytes, BE u16). |
| `msg_type` | `0x05`. |
| `nonce` | 8-byte random nonce (generated by node per transmission; **not** a sequence number вҖ” see note below). |

> **Nonce semantics:**  `PEER_REQUEST`/`PEER_ACK` use a **random** nonce, not the gateway-assigned sequence number used by post-WAKE messages (`COMMAND`, `DATA`, etc.).  These messages are sent before any session is established, so no sequence counter exists.  Gateway implementations MUST NOT apply the normal post-WAKE sequence-number anti-replay checks to `msg_type` `0x05`/`0x84` frames.  See [protocol.md](protocol.md) В§3.1 for the standard nonce semantics.
>
> **Note:** These message types (`0x05`/`0x84`) extend the base protocol defined in `protocol.md`.  `protocol.md` will be updated to include them in a follow-up PR.

**CBOR payload** (map with integer keys):

```cbor
{ 1: bstr }    ; key 1 = encrypted_payload (opaque blob from phone)
```

CBOR integer keys are scoped per `msg_type` вҖ” key `1` in a `PEER_REQUEST` is unrelated to key `1` in a `WAKE` message.  This follows the same convention as the existing protocol ([protocol.md](protocol.md)).

**HMAC**: HMAC-SHA256 over (header вҖ– cbor_payload), keyed with the node PSK.

The node transmits `PEER_REQUEST` on each boot cycle until it receives `PEER_ACK(0x00)`.  Retransmission interval follows the normal wake cycle schedule (default 60 seconds).

### 7.2  PEER_ACK (msg_type 0x84, Gateway вҶ’ Node)

```
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ Header (11 bytes)                       в”Ӯ CBOR payload  в”Ӯ HMAC   в”Ӯ
в”Ӯ key_hint[2] вҖ– msg_type[1] вҖ– nonce[8]   в”Ӯ               в”Ӯ [32]   в”Ӯ
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

| Header field | Value |
|-------------|-------|
| `key_hint` | Node's key_hint (echoed back). |
| `msg_type` | `0x84`. |
| `nonce` | Echoes the `nonce` from the corresponding PEER_REQUEST (same as the nonce-echo pattern used in COMMAND). |

**CBOR payload**:

```cbor
{
  1: uint,    ; status (0 = registered)
  2: bstr     ; registration_proof (32 bytes)
}
```

The only status value sent is `0` (registered).  All error conditions result in silent discard вҖ” no `PEER_ACK` is sent.  This is consistent with the radio protocol's silent-discard security model ([protocol.md В§8](protocol.md#8--error-handling)).

The `registration_proof` field is `HMAC-SHA256(node_psk, "sonde-peer-ack-v1" вҖ– encrypted_payload)` вҖ” an HMAC binding the acknowledgement to the exact encrypted payload.  Both the node and the gateway can compute this value (both have the node PSK and the encrypted_payload bytes).

A BLE eavesdropper observing `NODE_PROVISION` captures both `node_psk` and `encrypted_payload`.  This constitutes a **node PSK compromise** вҖ” the attacker can race the legitimate node to register first, or impersonate the node on ESP-NOW after registration.  The `node_id` uniqueness check (step 10 in В§7.3) and timestamp tolerance (Вұ86400 s, step 9) limit replay and duplication but do not prevent a race.  The primary mitigation is using a **MITM-resistant BLE pairing method** (Passkey Entry or Numeric Comparison) to prevent eavesdropping.  Just Works is acceptable only in low-threat environments where physical proximity provides adequate assurance.

**Node verification of PEER_ACK:**

1. Verify frame HMAC over (header вҖ– cbor_payload) using node PSK.
2. Verify `nonce` matches the nonce sent in the PEER_REQUEST.
3. Compute expected `registration_proof` = `HMAC-SHA256(node_psk, "sonde-peer-ack-v1" вҖ– encrypted_payload)` using the stored encrypted payload from NVS.
4. If `registration_proof` does not match вҶ’ discard (potential forgery).
5. If all checks pass вҶ’ set registration-complete flag.
6. **The node retains the encrypted payload in NVS** until its first successful WAKE/COMMAND exchange with the gateway.  Only after the gateway responds to a normal WAKE message (proving the registration was actually accepted) does the node erase the encrypted payload.  This ensures that even if a forged PEER_ACK is accepted, the node can recover by retransmitting the PEER_REQUEST if the subsequent WAKE fails.

**HMAC**: HMAC-SHA256 over (header вҖ– cbor_payload), keyed with the node PSK.

### 7.3  Gateway processing of PEER_REQUEST

The gateway performs these steps **in order**.  Failure at any step causes **silent discard** вҖ” no `PEER_ACK` is sent.  This is consistent with the radio protocol's error handling ([protocol.md В§8](protocol.md#8--error-handling)):

1. **Parse header.** Extract key_hint, verify msg_type = 0x05.  If not вҶ’ discard.
2. **Unknown key_hint.** The node is not yet registered вҖ” the gateway cannot verify the HMAC yet.  Proceed tentatively.
3. **Parse CBOR.** Extract `encrypted_payload` (key 1, bstr).  If missing or malformed вҶ’ discard.
4. **Decrypt.**
   - Extract `eph_public[32] вҖ– nonce[12] вҖ– ciphertext[...]` from encrypted_payload.
   - Convert `gw_private_key` (Ed25519 seed) to X25519 private key (see В§5.5 step 1).
   - `shared_secret` = X25519(gw_x25519_private, eph_public).
   - `aes_key` = HKDF-SHA256(ikm = shared_secret, salt = gateway_id, info = `"sonde-node-pair-v1"`, len = 32).
   - Decrypt: `authenticated_request` = AES-256-GCM-Decrypt(key = aes_key, nonce = nonce, ciphertext = ciphertext, aad = gateway_id).
   - If decryption fails (bad GCM tag) вҶ’ discard.
5. **Parse authenticated_request.**
   - `phone_key_hint` = bytes[0..2] (BE u16).
   - `cbor_bytes` = bytes[2 .. lenвҲ’32].
   - `phone_hmac` = bytes[lenвҲ’32 .. len].
6. **Verify phone HMAC.**
   - Look up all phone PSKs matching `phone_key_hint` in the phone trust store (key_hint is a 16-bit non-unique lookup hint; collisions are possible).
   - For each matching PSK that is not revoked, compute expected = HMAC-SHA256(phone_psk, cbor_bytes).
   - If `phone_hmac` matches any candidate вҶ’ accept that PSK.
   - If no candidate matches (or all are revoked) вҶ’ discard.
7. **Parse PairingRequest CBOR.**  Extract node_id, node_key_hint, node_psk, rf_channel, sensors, timestamp.
8. **Verify frame HMAC.**
   - Using the extracted `node_psk`, compute HMAC over (header вҖ– cbor_payload) from the original ESP-NOW frame.
   - If HMAC mismatch вҶ’ discard.
9. **Verify timestamp.** If `|now вҲ’ timestamp| > 86400` вҶ’ discard.
10. **Check node_id uniqueness.** If node_id already registered вҶ’ discard.
11. **Verify key_hint consistency.** Frame header key_hint must match `node_key_hint` in the PairingRequest.  If mismatch вҶ’ discard.
12. **Register node.** Store (node_id, node_key_hint, node_psk, rf_channel, sensors, registered_by = phone_key_hint).
13. **Send PEER_ACK(0).** Compute `registration_proof` = `HMAC-SHA256(node_psk, "sonde-peer-ack-v1" вҖ– encrypted_payload)`.  Build CBOR payload `{ 1: 0, 2: registration_proof }`.  HMAC the frame with the node PSK.  The `nonce` field echoes the `nonce` from the PEER_REQUEST header.

### 7.4  HMAC bootstrap rationale

The PEER_REQUEST frame is HMAC'd with the node PSK (step 8), but the gateway doesn't know the node PSK until after decrypting the inner payload (step 4вҖ“7).  This is safe because:

- The encrypted payload can only be decrypted by the gateway (ECDH confidentiality).
- The phone HMAC proves the PSK was issued by an authorized phone (authentication).
- The frame HMAC proves the transmitting node holds the same PSK (proof of provisioning).
- An attacker who intercepts the node PSK over BLE cannot forge the encrypted payload (needs phone PSK + gateway public key).

## 8  Node firmware behavior

### 8.1  Boot priority

On reset, the node checks conditions in this order:

| Priority | Condition | Action |
|----------|-----------|--------|
| 1 | USB-CDC detected | Enter legacy USB pairing mode ([pairing-protocol.md](pairing-protocol.md)). |
| 2 | No PSK in NVS **or** pairing button held (GPIO LOW for вүҘ500ms after reset) | Enter BLE pairing mode (В§8.2). |
| 3 | PSK stored, registration-complete flag NOT set | Send PEER_REQUEST over ESP-NOW (В§8.3). |
| 4 | PSK stored, registration-complete flag set | Normal WAKE cycle. |

### 8.2  BLE pairing mode

1. Start BLE stack, register Node Provisioning Service (В§3.3).
2. Begin advertising with service UUID `0000FE50-...`.  Advertising name: `"sonde-XXXX"` where `XXXX` is the last 4 hex digits of the MAC address.
3. On BLE connection: negotiate MTU вүҘ 247, accept LESC Just Works pairing.
4. On `NODE_PROVISION` write:
   a. Parse fields per В§6.6.
   b. If already paired and pairing button was NOT held вҶ’ respond `NODE_ACK(0x01)`.
   c. If pairing button was held вҶ’ erase existing PSK first (factory reset), then proceed.
   d. Write to NVS: `node_psk`, `node_key_hint`, `rf_channel`, `encrypted_payload`.
   e. Clear registration-complete flag.
   f. Respond `NODE_ACK(0x00)`.
5. Remain in BLE pairing mode (allows re-provision on error).
6. On BLE disconnect вҶ’ reboot.

### 8.2.1  Factory reset via BLE

There is no dedicated BLE factory-reset message.  To factory-reset a node via BLE:

1. Hold the pairing button during boot to enter BLE pairing mode.
2. Send a new `NODE_PROVISION` вҖ” step 4c above erases the existing PSK and all persistent state before writing the new credentials.

For explicit factory reset without re-pairing, use the USB `RESET_REQUEST` command ([pairing-protocol.md](pairing-protocol.md) В§4.3).

### 8.3  Post-provision boot (registration pending)

1. Initialize ESP-NOW on the stored `rf_channel`.
2. Load `encrypted_payload` from NVS.
3. Build and transmit `PEER_REQUEST` frame (В§7.1).
4. Listen for `PEER_ACK` for 10 seconds.
5. On `PEER_ACK(0)` with valid HMAC, nonce echo, and registration_proof:
   - Set registration-complete flag in NVS.
   - Retain `encrypted_payload` in NVS (erased after first successful WAKE вҖ” see В§8.3.1).
   - Proceed to normal WAKE cycle.
6. On timeout, HMAC failure, or proof mismatch:
   - Enter deep sleep.
   - Retry on next wake (default 60 seconds).

#### 8.3.1  Deferred payload erasure

After the registration-complete flag is set, the node attempts a normal WAKE cycle.  If the gateway responds with a valid COMMAND (proving the node is registered and the gateway knows its PSK), the node erases the `encrypted_payload` from NVS.

If the WAKE fails (no response or HMAC failure), the node clears the registration-complete flag and reverts to sending `PEER_REQUEST` on the next boot.  This self-healing behavior ensures that a forged `PEER_ACK` (from an attacker who intercepted the node PSK) does not permanently brick the node.

### 8.4  NVS layout

| NVS key | Type | Size | Description |
|---------|------|------|-------------|
| `magic` | u32 | 4 | Presence marker (`0x534F4E44` вҖ” big-endian ASCII "SOND"). |
| `key_hint` | u32 | 4 | Node key_hint (stored as u32, used as u16). |
| `psk` | blob | 32 | Node PSK. |
| `channel` | u32 | 4 | RF channel (1вҖ“13). |
| `peer_payload` | blob | variable | Encrypted pairing payload.  Retained until the first successful WAKE/COMMAND exchange, then erased. |
| `reg_complete` | u32 | 4 | Registration-complete flag (1 = registered). |
| `interval` | u32 | 4 | Wake cycle interval (seconds). |
| `active_p` | u32 | 4 | Active program partition (0 or 1). |
| `prog_a` | blob | вүӨ4096 | Program partition A. |
| `prog_b` | blob | вүӨ4096 | Program partition B. |

## 9  Security analysis

### 9.1  Threat model summary

| Threat | Mitigated by |
|--------|-------------|
| Rogue gateway impersonation (Phase 1) | Gateway signs phone's challenge nonce with Ed25519 private key; phone verifies before proceeding (В§5.3). |
| Unauthorized phone registration | Registration window must be opened by operator action; gateway rejects `REGISTER_PHONE` when closed (В§5.4.1). |
| Rogue phone registers a node | Phone HMAC verification (step 6 in В§7.3).  Attacker needs a valid phone PSK. |
| Compromised phone PSK | Gateway operator revokes the phone PSK.  Previously registered nodes remain valid. |
| BLE passive eavesdrop captures node PSK | `NODE_PROVISION` sends both `node_psk` and `encrypted_payload` over BLE.  A BLE eavesdropper captures all pairing material.  Mitigated by one-time-use of the encrypted payload (`node_id` uniqueness, step 10 in В§7.3) and timestamp tolerance (Вұ86400 s, step 9). |
| BLE active MITM captures node PSK | Same as passive eavesdrop.  For high-assurance deployments, use BLE Passkey Entry or Numeric Comparison pairing to prevent MITM.  Just Works is acceptable for low-threat environments. |
| Forged PEER_ACK | `registration_proof` = HMAC over encrypted payload (В§7.2) raises the bar.  Deferred payload erasure (В§8.3.1) ensures the node self-heals if a forged ACK is accepted вҖ” it reverts to PEER_REQUEST if the subsequent WAKE fails. |
| Replay of PEER_REQUEST | Timestamp check (Вұ24h) + node_id uniqueness (step 9вҖ“10 in В§7.3). |
| Phone A reads Phone B's pairing requests | Impossible вҖ” each encrypted with gateway public key, authenticated with different phone PSK. |
| Node impersonation after registration | HMAC-SHA256 on every ESP-NOW frame (existing radio protocol). |

### 9.2  BLE link security

BLE LESC Just Works provides AES-CCM encrypted transport.  This is the **only** protection for the node PSK in transit (phone вҶ’ node).  An active MITM can defeat Just Works and capture both the `node_psk` and `encrypted_payload` from `NODE_PROVISION`, which is sufficient to craft a valid `PEER_REQUEST`.  The primary mitigations are the one-time-use nature of the encrypted payload (the gateway rejects duplicate `node_id` registrations, step 10 in В§7.3) and the PairingRequest timestamp tolerance (Вұ86400 s, step 9 in В§7.3).  The 120 s registration window applies only to Phase 1 phone registration (`REGISTER_PHONE` В§5.4.1), not to Phase 2 node registration.

For higher-security deployments, BLE Passkey Entry or Numeric Comparison could be used instead of Just Works (the protocol is agnostic to the BLE pairing method).

### 11.1  Encrypted payload size budget

The ESP-NOW maximum frame payload is 250 bytes.  The PEER_REQUEST frame budget:

```
Header:            11 bytes  (key_hint[2] + msg_type[1] + nonce[8])
CBOR overhead:      4 bytes  (1-pair map: 0xA1 + key + bstr-2 header = 4 bytes for payloads вүӨ 255B)
HMAC:              32 bytes
в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
Available for encrypted_payload:  203 bytes (exact)
```

The constant `MAX_ENCRYPTED_PAYLOAD` is set to **202 bytes** (conservative, reserving 1 byte for possible future CBOR keys).  The `protocol.md` payload budget of 207 bytes (250 вҲ’ 11 вҲ’ 32) is 4 bytes larger because it does not include CBOR framing overhead.

The encrypted payload contains:

```
eph_public:        32 bytes
nonce:             12 bytes
GCM tag:           16 bytes
phone_key_hint:     2 bytes
phone_hmac:        32 bytes
CBOR PairingRequest: variable
в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
Fixed overhead:    94 bytes
Available for CBOR: ~108 bytes
```

A minimal PairingRequest (20-char node_id, no sensors) is ~80 bytes CBOR.  This fits.  Large sensor arrays may require splitting across multiple PEER_REQUEST frames (future extension) or limiting sensor metadata.
